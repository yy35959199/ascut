## 统一解释框架

```
角色提示的效果 = f(任务类型 × 模型优化程度 × 提示粒度)
```


| 维度            | 规律                                            |
| ------------- | --------------------------------------------- |
| **任务类型**      | 对齐相关任务（写作、安全）→ 持续受益；预训练知识任务（MMLU、数学、编程）→ 持续受损 |
| **模型优化**      | 指令调优越深，双向效果越强；推理蒸馏模型遵循蒸馏集分布，安全能力被彻底抹除         |
| **提示粒度**      | 在受益方向上越长越好；在受损方向上越长越坏                         |
| **角色位置**      | 高优化模型对系统提示位置敏感；低优化模型位置无关                      |
| **专家 vs. 随机** | 指令调优模型中专家优于随机；推理模型中专家劣于随机                     |

## 一、模型选择：`deepseek-chat` vs `deepseek-reasoner`

两者都是DeepSeek-V3.2，区别仅在于是否开启思考模式。

### 逐层分析

**理解R1（主旨识别 + 语义分段）**

- 任务本质：全局语义压缩 + 主题边界检测
- 这是**模式识别型任务**——模型需要在长序列上做注意力聚合，提取高层语义，识别主题转换点
- 不需要显式的多步推理链。"这段在讲A，那段在讲B"这种判断是单次前向传播可以完成的语义匹配
- Reasoner的推理链在这里没有清晰的推理步骤可以展开，反而会消耗大量token（推理链token也计费），增加延迟
- **推荐：`deepseek-chat`（非思考模式）**

**理解R2（纠错）**

- 任务本质：对R1输出进行一致性检查和修正
- 这里需要在"R1的分块结果"和"原文"之间做**比对推理**——"R1说句15-32是关于主题B的，但句20-25实际上在讲主题C"
- 这种跨两个文本的比对、发现不一致、提出修正，具有**多步推理**特征
- 但复杂度有限——不是数学证明级别的推理链，更像是"逐条核对"
- **推荐：`deepseek-chat` 优先尝试**。如果发现纠错质量不足（漏检率高），再切换到 `deepseek-reasoner`。理由是纠错的推理深度有限，chat模型大概率够用，而reasoner的token成本和延迟显著更高

**决策层（逐块精简）**

- 任务本质：对每个句子做二元判断——保留还是删除
- 这是一个**逐句判别任务**，判断依据是"该句与主旨的语义相关性"
- 单次前向传播完全可以完成这种语义相关性判断
- Reasoner在这里反而危险：推理链可能导致**过度思考**（overthinking），对边界情况反复纠结，最终做出不一致的判断
- **推荐：`deepseek-chat`（非思考模式）**

**审核层（对照审核）**

- 任务本质：三方比对（主旨 vs 原文块 vs 精简块），检查遗漏和冗余
- 这是整个流程中**推理需求最高**的步骤：
  - 需要逐句扫描原文，检查是否有与主旨相关但被删掉的句子（遗漏检测）
  - 需要逐句扫描精简结果，检查是否有与主旨无关但被保留的句子（冗余检测）
  - 需要在三个文本之间来回跳转比对
- 这种"多文本交叉比对 + 逐条判断 + 汇总结论"的模式，是Reasoner的典型优势场景
- **推荐：`deepseek-reasoner`（思考模式）**

### 汇总


| 层    | 推荐模型                                          | 理由                     |
| ---- | --------------------------------------------- | ---------------------- |
| 理解R1 | `deepseek-chat`                               | 语义压缩+边界检测，模式识别型，不需要推理链 |
| 理解R2 | `deepseek-chat`（优先），效果不足则 `deepseek-reasoner` | 纠错推理深度有限，chat大概率够用     |
| 决策层  | `deepseek-chat`                               | 逐句二元判别，推理链反而有害         |
| 审核层  | `deepseek-reasoner`                           | 三方交叉比对，多步推理需求最高        |

```json

```

| 层    | 是否用得上 | 说明                                                                     |
| ---- | ----- | ---------------------------------------------------------------------- |
| 理解R1 | ❌ 不用  | 不需要思考模式                                                                |

| 决策层  | ❌ 不用  | 不需要思考模式                                                                |
| 审核层  | ✅ 用   | 开启思考模式。实际上用 `model="deepseek-reasoner"` 和 `thinking.type="enabled"` 等价 |


**工程价值**：统一用 `model="deepseek-chat"` + `thinking` 参数动态切换，代码更统一，不需要维护两套模型调用逻辑。

### 2.2 `response_format: json_object`（JSON输出模式）


| 层    | 是否用得上      | 说明                                        |
| ---- | ---------- | ----------------------------------------- |
| 理解R1 | ✅ **强烈推荐** | 输出是结构化数据（主旨 + 分块列表 + 各块概要），JSON格式便于下游程序解析 |

| 审核层  | ✅ **推荐**   | 输出是遗漏列表、冗余列表、判定结果                         |


**理由**：你的全部四层输出都是结构化数据，不是自由文本。JSON模式确保输出可被程序直接解析，避免正则提取的脆弱性。

### 2.8 上下文硬盘缓存

这个特性对你的场景**非常有价值**，而且是自动生效的。

**缓存命中条件**：后续请求与之前请求的**输入前缀**重复。

逐层分析缓存命中的可能性：

**理解R1 → 理解R2**

```
R1的messages:
  system: "你将收到一段视频转录文本..."
  user: [全文句子List]

R2的messages:
  system: "你将收到一段视频转录文本及其分析结果..."  ← 不同
  user: [全文句子List + R1输出]
```

system prompt不同，前缀不匹配，**缓存不命中**。

**但如果你这样设计**：

```
R1和R2共享相同的system prompt，把任务指令放在user message的开头：

R1的messages:
  system: [固定的通用指令]
  user: "任务：主旨识别与分段\n\n" + [全文句子List]

R2的messages:
  system: [固定的通用指令]  ← 相同！
  user: "任务：纠错\n\n" + [全文句子List] + [R1输出]
```

这样system prompt前缀匹配，**可以命中缓存**。但user message开头不同（"任务：主旨识别" vs "任务：纠错"），所以实际上还是不命中。

**真正能命中缓存的场景是决策层的逐块处理**：

```
决策层-块1:
  system: [精简指令]
  user: "主旨：{主旨}\n\n块1内容：..."

决策层-块2:
  system: [精简指令]          ← 相同！
  user: "主旨：{主旨}\n\n"    ← 前缀相同！
        "块2内容：..."        ← 从这里开始不同
```

**system prompt + user message中"主旨：{主旨}"的部分是完全相同的前缀**，可以命中缓存。如果主旨描述 + system prompt超过64 tokens（几乎肯定），缓存就会生效。


| 层       | 缓存命中可能性    | 说明                           |
| ------- | ---------- | ---------------------------- |
| 理解R1    | ❌ 首次请求，无缓存 | —                            |
| 理解R2    | ⚠️ 低       | 除非刻意设计前缀一致                   |
| 决策层（逐块） | ✅ **高**    | 多个块共享相同的system + 主旨前缀，第2块起命中 |
| 审核层（逐块） | ✅ **高**    | 同理，多个块共享相同的system + 主旨前缀     |


**工程建议**：为了最大化缓存命中率，决策层和审核层的prompt应该这样设计：

```python
# 所有块共享的前缀（会被缓存）
system_prompt = "你将对文本块进行精简..."  # 固定
shared_prefix = f"主旨：{topic}\n\n精简规则：...\n\n"  # 固定

# 每块不同的部分（不被缓存）
for block in blocks:
    user_message = shared_prefix + f"当前块内容：\n{block['text']}"
    # 调用API
```

**成本影响**：缓存命中的token按0.1元/百万tokens计费，未命中按1元/百万tokens——**10倍差价**。如果你的视频转录有20个块，决策层的19次请求（第2块起）都能在前缀部分命中缓存，成本显著降低。

### 2.12 多轮对话（手动拼接历史）


| 层       | 是否用得上      | 说明                        |
| ------- | ---------- | ------------------------- |
| 理解R1→R2 | ✅ **核心机制** | R2需要看到R1的输出才能纠错。两种实现方式见下文 |
| 决策层     | ❌          | 每块独立处理，不需要对话历史            |
| 审核层     | ❌          | 每块独立审核，不需要对话历史            |


**R1→R2的两种实现方式**：

**方式A：真正的多轮对话**

```python
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": "请分析以下文本的主旨和分段：\n" + full_text},
]
r1_response = call_api(messages)  # R1

messages.append({"role": "assistant", "content": r1_response})
messages.append({"role": "user", "content": "请检查上述分析是否存在以下问题：1. 分块边界是否准确..."})
r2_response = call_api(messages)  # R2
```

**方式B：单轮调用，把R1输出塞进R2的user message**

```python
r2_messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": f"以下是对文本的初步分析：\n{r1_output}\n\n原文：\n{full_text}\n\n请纠错..."},
]
```

**推荐方式A**，理由：

1. 多轮对话的前缀（system + R1的user message）可以命中缓存
2. 语义更自然，模型更容易理解"这是对上一轮的修正"

## 三、关于审核层的Checklist

你问"审核层是不是最好加个checklist"——**是的，强烈建议**。

从论文的角度，checklist的作用等价于**将隐式的判断标准外显化为显式的任务指令**，这与"不用角色提示，改用任务约束"的原则完全一致。

**推荐的Checklist设计**：

```json
{
  "checklist": [
    {
      "id": "C1",
      "check": "遗漏检测：原文中是否有与主旨直接相关、但被精简掉的句子？",
      "severity": "high"
    },
    {
      "id": "C2", 
      "check": "冗余检测：精简结果中是否保留了与主旨无关的句子？",
      "severity": "medium"
    },
    {
      "id": "C3",
      "check": "语义连贯性：精简后的句子序列是否仍然逻辑连贯，不存在突兀的跳跃？",
      "severity": "medium"
    },
    {
      "id": "C4",
      "check": "重复检测：精简结果中是否仍存在语义重复的句子？",
      "severity": "low"
    },
    {
      "id": "C5",
      "check": "边界完整性：是否有句子被截断或拆分？",
      "severity": "low"
    }
  ]
}
```

**审核层的输出schema**：

```json
{
  "checks": [
    {"id": "C1", "pass": false, "issues": [{"sentence_id": 20, "reason": "该句讨论了核心论点X，与主旨直接相关但被删除"}]},
    {"id": "C2", "pass": true, "issues": []},
    {"id": "C3", "pass": true, "issues": []},
    {"id": "C4", "pass": true, "issues": []},
    {"id": "C5", "pass": true, "issues": []}
  ],
  "overall": "fail",
  "reasoning_summary": "C1未通过：句子20包含核心论点X的关键论据，不应删除"
}
```

**为什么checklist对reasoner特别有效**：Reasoner的推理链需要"锚点"来组织推理步骤。Checklist的每一条就是一个天然的锚点——模型会在推理链中逐条检查，而不是漫无目的地"想"。这降低了推理链偏移（reasoning drift）的风险。

## 四、最终架构汇总


| 层    | 模型                       | 关键API特性                                                                | temperature | 输出格式              |
| ---- | ------------------------ | ---------------------------------------------------------------------- | ----------- | ----------------- |
| 理解R1 | `deepseek-chat`          | `response_format: json_object`                                         | 0           | JSON（主旨+分块）       |
| 理解R2 | `deepseek-chat`（多轮对话接R1） | `response_format: json_object`，利用缓存                                    | 0           | JSON（修正后分块）       |
| 决策层  | `deepseek-chat`（逐块并发）    | `response_format: json_object`，缓存命中（共享主旨前缀）                            | 0           | JSON（保留/删除列表）     |
| 审核层  | `deepseek-reasoner`（逐块）  | `response_format: json_object`，`reasoning_content`记录推理链，缓存命中，checklist | N/A（不生效）    | JSON（checklist判定） |


**成本优化要点**：

1. 理解R1/R2只调用2次，成本固定
2. 决策层和审核层是逐块调用，块数越多调用越多——但缓存机制让第2块起的前缀部分按1/10价格计费
3. 审核层用reasoner，推理链token会显著增加completion_tokens——这是主要成本来源。可以通过`max_tokens`限制推理链长度来控制成本，但要注意不能设太小导致推理被截断

## 二、流式 vs 非流式的决策

### 从用户体验角度

如果未来要做GUI，流式是**几乎必须的**，原因：

1. **长文本处理耗时长**：一个20分钟的直播录屏，ASR转录可能有500-1000句。理解层的单次调用可能需要10-30秒（取决于输入长度和模型推理时间）。非流式调用期间，用户看到的是一个完全无反馈的等待界面
2. **审核层的推理链非常长**：reasoner的thinking过程可能输出数千tokens，非流式需要全部生成完才返回，延迟可能达到30-60秒
3. **逐块处理的进度感知**：决策层和审核层是逐块处理的，用户需要看到"正在处理第3/15块"这种进度

### 从工程角度

流式和非流式在DeepSeek API中的差异：

**非流式**：

```python
response = client.chat.completions.create(stream=False, ...)
content = response.choices[0].message.content
reasoning = response.choices[0].message.reasoning_content  # reasoner only
```

**流式**：

```python
stream = client.chat.completions.create(stream=True, ...)
for chunk in stream:
    delta = chunk.choices[0].delta
    # chat模式：delta.content 逐token到达
    # reasoner模式：delta.reasoning_content 先到达，然后 delta.content 到达
```

**流式的额外复杂度**：

1. 需要拼接chunks为完整JSON（流式返回的是JSON片段，不是完整JSON）
2. reasoner流式返回中，`reasoning_content`和`content`是**分阶段到达**的——先所有reasoning chunks，再所有content chunks
3. JSON模式 + 流式：每个chunk的`delta.content`是JSON字符串的片段，需要累积后再`json.loads`

### 结论

**底层方法全部实现为流式，通过回调/生成器暴露给上层**。非流式场景只需在上层收集完整结果即可——流式是非流式的超集。

```
流式方法（底层） → 非流式包装（便捷方法）
                 → GUI流式展示（直接使用）
```

## 三、方法设计

### 设计原则

1. **底层方法按"模式×轮次"组织**，不按业务层组织——业务层是底层方法的组合
2. **流式作为默认**，非流式是流式的包装
3. **JSON解析统一处理**，不在每个业务层重复
4. **reasoning_content的提取在底层方法中完成**，业务层不需要关心

### 方法拆分

```
层次结构：

┌─────────────────────────────────────────────────┐
│  业务层方法（Pipeline Orchestrator）              │
│  run_understanding_r1()                          │
│  run_understanding_r2()                          │
│  run_decision(blocks)                            │
│  run_review(blocks)                              │
│  run_pipeline(text) ← 串联全流程                  │
├─────────────────────────────────────────────────┤
│  中间层（流式JSON收集 + 回调分发）                 │
│  stream_chat_json()                              │
│  stream_reasoner_json()                          │
├─────────────────────────────────────────────────┤
│  底层（API调用封装）                              │
│  _call_stream()        ← 唯一的API调用点          │
└─────────────────────────────────────────────────┘
```

### 底层：唯一的API调用点

只需要**一个**底层方法，所有差异通过参数控制：

```python
from openai import OpenAI
from typing import Generator, Optional
from dataclasses import dataclass

@dataclass
class StreamChunk:
    """流式输出的统一数据结构"""
    chunk_type: str  # "reasoning" | "content" | "done" | "error"
    text: str        # 当前chunk的文本片段
    usage: Optional[dict] = None  # 仅在 "done" 时有值

def _call_stream(
    client: OpenAI,
    messages: list[dict],
    *,
    thinking: bool = False,
    json_mode: bool = True,
    temperature: float = 0.0,
    max_tokens: int = 8192,
) -> Generator[StreamChunk, None, None]:
    """
    唯一的API调用方法。所有上层方法都通过此方法调用DeepSeek API。
    
    - thinking=False → deepseek-chat 非思考模式
    - thinking=True  → deepseek-chat 思考模式（等价于deepseek-reasoner）
    - 始终使用流式输出
    """
    kwargs = {
        "model": "deepseek-chat",
        "messages": messages,
        "stream": True,
        "stream_options": {"include_usage": True},  # 末尾返回usage
        "max_tokens": max_tokens,
    }
    
    if json_mode:
        kwargs["response_format"] = {"type": "json_object"}
    
    if thinking:
        kwargs["extra_body"] = {"thinking": {"type": "enabled"}}
        # 思考模式下 temperature/top_p 不生效，不设置
    else:
        kwargs["temperature"] = temperature
    
    try:
        stream = client.chat.completions.create(**kwargs)
        for chunk in stream:
            # 最后一个chunk携带usage，choices可能为空
            if not chunk.choices:
                if hasattr(chunk, 'usage') and chunk.usage:
                    yield StreamChunk(
                        chunk_type="done",
                        text="",
                        usage=chunk.usage.model_dump() if chunk.usage else None
                    )
                continue
            
            delta = chunk.choices[0].delta
            
            # reasoner模式：先输出reasoning_content，再输出content
            if hasattr(delta, 'reasoning_content') and delta.reasoning_content:
                yield StreamChunk(chunk_type="reasoning", text=delta.reasoning_content)
            
            if delta.content:
                yield StreamChunk(chunk_type="content", text=delta.content)
                
    except Exception as e:
        yield StreamChunk(chunk_type="error", text=str(e))
```

**为什么只需要一个底层方法**：

- chat vs reasoner的区别只是`thinking`参数——`True/False`
- 流式是统一的，非流式通过上层收集实现
- JSON模式只是一个参数开关
- temperature在thinking模式下自动跳过

### 中间层：流式JSON收集 + 回调分发

这里需要**两个方法**，区别在于是否处理`reasoning_content`：

```python
import json
from typing import Callable, Optional
from dataclasses import dataclass

@dataclass
class ChatResult:
    """非思考模式的返回结果"""
    content: dict          # 解析后的JSON
    raw_content: str       # 原始JSON字符串
    usage: Optional[dict]

@dataclass  
class ReasonerResult:
    """思考模式的返回结果"""
    content: dict          # 解析后的JSON
    raw_content: str       # 原始JSON字符串
    reasoning: str         # 完整的推理链文本
    usage: Optional[dict]


def stream_chat_json(
    client: OpenAI,
    messages: list[dict],
    *,
    temperature: float = 0.0,
    max_tokens: int = 8192,
    on_content_chunk: Optional[Callable[[str], None]] = None,
) -> ChatResult:
    """
    中间层方法：chat模式 + JSON输出 + 流式收集
    
    on_content_chunk: 每收到一个content chunk时的回调
                      GUI层可以通过此回调实时展示生成进度
    """
    content_parts = []
    usage = None
    
    for chunk in _call_stream(
        client, messages,
        thinking=False,
        json_mode=True,
        temperature=temperature,
        max_tokens=max_tokens,
    ):
        if chunk.chunk_type == "content":
            content_parts.append(chunk.text)
            if on_content_chunk:
                on_content_chunk(chunk.text)
        elif chunk.chunk_type == "done":
            usage = chunk.usage
        elif chunk.chunk_type == "error":
            raise RuntimeError(f"API调用失败: {chunk.text}")
    
    raw = "".join(content_parts)
    return ChatResult(
        content=json.loads(raw),
        raw_content=raw,
        usage=usage,
    )


def stream_reasoner_json(
    client: OpenAI,
    messages: list[dict],
    *,
    max_tokens: int = 16384,
    on_reasoning_chunk: Optional[Callable[[str], None]] = None,
    on_content_chunk: Optional[Callable[[str], None]] = None,
) -> ReasonerResult:
    """
    中间层方法：reasoner模式 + JSON输出 + 流式收集
    
    on_reasoning_chunk: 每收到一个reasoning chunk时的回调
                        GUI层可以展示"模型正在思考..."的实时过程
    on_content_chunk:   每收到一个content chunk时的回调
    """
    reasoning_parts = []
    content_parts = []
    usage = None
    
    for chunk in _call_stream(
        client, messages,
        thinking=True,
        json_mode=True,
        max_tokens=max_tokens,
    ):
        if chunk.chunk_type == "reasoning":
            reasoning_parts.append(chunk.text)
            if on_reasoning_chunk:
                on_reasoning_chunk(chunk.text)
        elif chunk.chunk_type == "content":
            content_parts.append(chunk.text)
            if on_content_chunk:
                on_content_chunk(chunk.text)
        elif chunk.chunk_type == "done":
            usage = chunk.usage
        elif chunk.chunk_type == "error":
            raise RuntimeError(f"API调用失败: {chunk.text}")
    
    raw = "".join(content_parts)
    return ReasonerResult(
        content=json.loads(raw),
        raw_content=raw,
        reasoning="".join(reasoning_parts),
        usage=usage,
    )
```

**为什么拆成两个而不是一个**：

虽然可以用一个方法 + `thinking`参数来统一，但返回类型不同（`ChatResult` vs `ReasonerResult`），回调也不同（reasoner多一个`on_reasoning_chunk`）。强行合并会导致：

- 返回类型需要用Union或Optional字段，调用方需要做类型判断
- 回调参数混乱

拆成两个方法，类型签名清晰，调用方不需要做任何条件判断。

### 业务层：Pipeline方法

```python
from typing import Callable, Optional
from dataclasses import dataclass
import asyncio  # 决策层/审核层的并发用

@dataclass
class PipelineCallbacks:
    """GUI层注入的回调集合"""
    on_stage_start: Optional[Callable[[str], None]] = None      # "理解R1" / "决策层-块3"
    on_content_chunk: Optional[Callable[[str], None]] = None     # 实时文本
    on_reasoning_chunk: Optional[Callable[[str], None]] = None   # 审核层推理链
    on_stage_complete: Optional[Callable[[str, dict], None]] = None  # 阶段完成+结果
    on_block_progress: Optional[Callable[[int, int], None]] = None   # 块进度 (3, 15)


def run_understanding_r1(
    client: OpenAI,
    sentences: list[str],
    callbacks: PipelineCallbacks = PipelineCallbacks(),
) -> tuple[ChatResult, list[dict]]:
    """
    理解层第一轮：主旨识别 + 语义分段
    
    返回：(ChatResult, messages_history)
    messages_history 传给R2实现多轮对话
    """
    if callbacks.on_stage_start:
        callbacks.on_stage_start("理解R1")
    
    system_prompt = """你将收到一段视频转录文本（以编号句子列表形式）。请完成：
1. 用一句话概括全文主旨
2. 将全文按语义主题拆分为若干连续的文本块，标注起止句号和概要

输出JSON格式：
{"topic": "...", "blocks": [{"block_id": 1, "start": 1, "end": 15, "summary": "..."}]}"""
    
    text_input = "\n".join(f"[{i+1}] {s}" for i, s in enumerate(sentences))
    
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": text_input},
    ]
    
    result = stream_chat_json(
        client, messages,
        on_content_chunk=callbacks.on_content_chunk,
    )
    
    # 保留对话历史供R2使用
    messages.append({"role": "assistant", "content": result.raw_content})
    
    if callbacks.on_stage_complete:
        callbacks.on_stage_complete("理解R1", result.content)
    
    return result, messages


def run_understanding_r2(
    client: OpenAI,
    messages_history: list[dict],
    sentences: list[str],
    callbacks: PipelineCallbacks = PipelineCallbacks(),
) -> ChatResult:
    """
    理解层第二轮：纠错（多轮对话，接R1的历史）
    """
    if callbacks.on_stage_start:
        callbacks.on_stage_start("理解R2")
    
    correction_prompt = """请检查上述分析是否存在以下问题并修正：
1. 分块边界是否准确？是否有主题切换点被遗漏？
2. 各块概要是否准确反映了该块的实际内容？
3. 全文主旨是否准确？

输出修正后的完整JSON（格式同上）。如无需修正，原样输出。"""
    
    messages_history.append({"role": "user", "content": correction_prompt})
    
    result = stream_chat_json(
        client, messages_history,
        on_content_chunk=callbacks.on_content_chunk,
    )
    
    if callbacks.on_stage_complete:
        callbacks.on_stage_complete("理解R2", result.content)
    
    return result


def run_decision_block(
    client: OpenAI,
    topic: str,
    block: dict,
    sentences: list[str],
    callbacks: PipelineCallbacks = PipelineCallbacks(),
) -> ChatResult:
    """
    决策层：处理单个块
    """
    system_prompt = """你将收到一段文本块及其所属文章的主旨。
请仅保留与主旨直接相关的内容，删除：
- 与主旨无关的闲聊、跑题内容
- 语义重复的句子（保留表述最清晰的一句）
- 口头禅、填充语、无信息量的过渡句

输出JSON格式：
{"block_id": N, "retained": [句号列表], "removed": [{"id": N, "reason": "..."}]}"""
    
    # 构造共享前缀（利用缓存）：system + 主旨部分
    block_sentences = sentences[block["start"]-1 : block["end"]]
    block_text = "\n".join(
        f"[{block['start']+i}] {s}" for i, s in enumerate(block_sentences)
    )
    
    user_content = f"主旨：{topic}\n\n块{block['block_id']}内容：\n{block_text}"
    
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_content},
    ]
    
    return stream_chat_json(
        client, messages,
        on_content_chunk=callbacks.on_content_chunk,
    )


def run_review_block(
    client: OpenAI,
    topic: str,
    block: dict,
    sentences: list[str],
    decision_result: dict,
    callbacks: PipelineCallbacks = PipelineCallbacks(),
) -> ReasonerResult:
    """
    审核层：处理单个块（使用reasoner）
    """
    checklist = """请逐条检查：
C1-遗漏检测：原文中是否有与主旨直接相关但被删掉的句子？
C2-冗余检测：保留结果中是否有与主旨无关的句子？
C3-连贯性：保留的句子序列是否逻辑连贯？
C4-重复检测：保留结果中是否仍有语义重复？

输出JSON格式：
{"block_id": N, "checks": [{"id": "C1", "pass": bool, "issues": [...]}], "overall": "pass"/"fail"}"""
    
    block_sentences = sentences[block["start"]-1 : block["end"]]
    original_text = "\n".join(
        f"[{block['start']+i}] {s}" for i, s in enumerate(block_sentences)
    )
    
    retained_ids = set(decision_result.get("retained", []))
    retained_text = "\n".join(
        f"[{block['start']+i}] {s}" 
        for i, s in enumerate(block_sentences)
        if (block["start"]+i) in retained_ids
    )
    
    user_content = (
        f"主旨：{topic}\n\n"
        f"原文（块{block['block_id']}）：\n{original_text}\n\n"
        f"精简结果：\n{retained_text}"
    )
    
    messages = [
        {"role": "system", "content": checklist},
        {"role": "user", "content": user_content},
    ]
    
    return stream_reasoner_json(
        client, messages,
        max_tokens=16384,  # 推理链需要更大空间
        on_reasoning_chunk=callbacks.on_reasoning_chunk,
        on_content_chunk=callbacks.on_content_chunk,
    )
```

## 四、并发处理（决策层和审核层的逐块并发）

决策层和审核层的各块之间是**独立的**，可以并发执行。但需要注意：

1. **DeepSeek API的并发限制**：免费tier通常有QPS/QPM限制，需要控制并发度
2. **缓存命中率**：并发请求如果同时到达，缓存可能还没建立。**顺序发送前几个请求，等缓存建立后再并发**，可以提高命中率
3. **GUI进度展示**：并发时多个块同时产出chunks，需要区分来源

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor

async def run_decision_all_blocks(
    client: OpenAI,
    topic: str,
    blocks: list[dict],
    sentences: list[str],
    callbacks: PipelineCallbacks = PipelineCallbacks(),
    max_concurrency: int = 3,
) -> list[ChatResult]:
    """
    决策层：所有块的并发处理
    """
    semaphore = asyncio.Semaphore(max_concurrency)
    loop = asyncio.get_event_loop()
    executor = ThreadPoolExecutor(max_workers=max_concurrency)
    
    async def process_block(block: dict, index: int) -> ChatResult:
        async with semaphore:
            if callbacks.on_block_progress:
                callbacks.on_block_progress(index + 1, len(blocks))
            
            # OpenAI SDK是同步的，用executor包装
            result = await loop.run_in_executor(
                executor,
                lambda: run_decision_block(client, topic, block, sentences, callbacks)
            )
            return result
    
    tasks = [process_block(b, i) for i, b in enumerate(blocks)]
    results = await asyncio.gather(*tasks)
    return list(results)
```

**关于并发 + 流式 + GUI的交互**：

当多个块并发执行时，每个块的流式chunks会交错到达。GUI需要区分"这个chunk来自哪个块"。解决方案：

```python
# 方案：给每个块的回调注入block_id标识
def make_block_callbacks(block_id: int, gui_handler) -> PipelineCallbacks:
    return PipelineCallbacks(
        on_content_chunk=lambda text: gui_handler.update_block(block_id, text),
        on_reasoning_chunk=lambda text: gui_handler.update_reasoning(block_id, text),
    )
```

## 五、完整Pipeline串联

```python
async def run_full_pipeline(
    client: OpenAI,
    sentences: list[str],
    callbacks: PipelineCallbacks = PipelineCallbacks(),
) -> dict:
    """
    完整流程串联
    """
    # === 理解R1 ===
    r1_result, messages_history = run_understanding_r1(
        client, sentences, callbacks
    )
    topic = r1_result.content["topic"]
    blocks = r1_result.content["blocks"]
    
    # === 理解R2 ===
    r2_result = run_understanding_r2(
        client, messages_history, sentences, callbacks
    )
    topic = r2_result.content["topic"]      # 可能被修正
    blocks = r2_result.content["blocks"]    # 可能被修正
    
    # === 决策层（并发）===
    decision_results = await run_decision_all_blocks(
        client, topic, blocks, sentences, callbacks
    )
    
    # === 审核层（并发）===
    review_results = await run_review_all_blocks(
        client, topic, blocks, sentences,
        [r.content for r in decision_results],
        callbacks
    )
    
    # === 汇总 ===
    return {
        "topic": topic,
        "blocks": blocks,
        "decisions": [r.content for r in decision_results],
        "reviews": [r.content for r in review_results],
        "review_reasonings": [
            r.reasoning for r in review_results  # 审核推理链，供人工审核
        ],
    }
```

## 六、方法数量汇总


| 层次       | 方法                                                                    | 数量                 | 职责                                |
| -------- | --------------------------------------------------------------------- | ------------------ | --------------------------------- |
| **底层**   | `_call_stream()`                                                      | **1**              | 唯一API调用点，处理所有参数差异                 |
| **中间层**  | `stream_chat_json()`                                                  | **1**              | chat模式的流式JSON收集                   |
|          | `stream_reasoner_json()`                                              | **1**              | reasoner模式的流式JSON收集 + reasoning提取 |
| **业务层**  | `run_understanding_r1()`                                              | **1**              | 理解第一轮                             |
|          | `run_understanding_r2()`                                              | **1**              | 理解第二轮（多轮对话）                       |
|          | `run_decision_block()`                                                | **1**              | 决策层单块处理                           |
|          | `run_review_block()`                                                  | **1**              | 审核层单块处理                           |
|          | `run_decision_all_blocks()`                                           | **1**              | 决策层并发编排                           |
|          | `run_review_all_blocks()`                                             | **1**              | 审核层并发编排                           |
|          | `run_full_pipeline()`                                                 | **1**              | 完整流程串联                            |
| **数据结构** | `StreamChunk` / `ChatResult` / `ReasonerResult` / `PipelineCallbacks` | **4**              | 统一数据类型                            |
| **总计**   |                                                                       | **10个方法 + 4个数据结构** |                                   |

## 七、关键设计决策总结


| 决策点     | 选择                                  | 理由                                                 |
| ------- | ----------------------------------- | -------------------------------------------------- |
| 底层方法数量  | 1个                                  | chat/reasoner的差异仅在参数层面，不值得拆两个方法                    |
| 中间层方法数量 | 2个                                  | 返回类型不同（有/无reasoning），拆开后类型签名更清晰                    |
| 流式/非流式  | 底层全部流式                              | 流式是非流式的超集，GUI需要流式，非流式通过上层收集实现                      |
| model参数 | 统一用`deepseek-chat` + `thinking`参数切换 | 文档确认二者等价，统一model字段简化代码，也利于缓存（model是缓存key的一部分）「待验证」 |
| 并发框架    | asyncio + ThreadPoolExecutor        | OpenAI SDK是同步的，用executor包装为异步；semaphore控制并发度       |
| 回调机制    | Callbacks数据类 + 可选回调函数               | GUI层注入回调实现实时展示；无GUI时回调为None，零开销                    |
| 缓存优化    | system prompt + 主旨前缀保持一致            | 决策层/审核层的多块请求共享前缀，第2块起命中缓存（10倍差价）                   |
