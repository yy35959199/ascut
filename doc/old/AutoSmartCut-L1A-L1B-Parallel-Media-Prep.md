## 1. 文档定位与目标

- 本文是“L1A/L1B 解耦后进一步并行化”的设计文档，不替代 `AutoSmartCut-MVP-Mini.md` 的主契约叙事。
- 目标问题：当前总体耗时由 L2（LLM）和 L3（局部重编码+拼接）共同拉长，且两者多为串行执行。
- 目标方案：在 L1B 完成后，提前做与最终成片相关的媒体预处理，在 L2 执行期间并行推进，减少 L2 结束后的 L3临界路径。

### 1.1 优化目标

- 缩短 `T_end_to_end`（总墙钟时长）。
- 保持当前单清单契约与 L2 语义一致性，不引入“第二套正式层间协议”。
- 允许有限冗余计算，但要可观测、可回退、可评估收益。

### 1.2 非目标

- 不在本文中规定 smartcut 内部实现细节（后续可解耦）。
- 不改变 L2 决策语义（`current.keep_mask` 仍是主决策输入）。
- 不要求一次性实现全自动并行编排，可先从“两次命令 + 缓存复用”开始。

## 2. 现状基线（已实现）

> 以下为当前仓库可确认行为。

- L1 已拆分为：
  - L1A：`run_l1a_asr_only`，输出 `raw_text` + 仅含 `index/content` 的 `annotations`（时间字段为空）。
  - L1B：`run_l1b_align_only`，在不改 `index/content` 的前提下回填 `t_start/t_end/gap_after`。
- L2 仅依赖 `annotations -> tokens(index,text)`，因此可直接消费 L1A 结果。
- L3 当前要求 `annotations` 每句都具备 `t_start/t_end`，仅 L1A 不足以直接进 L3。
- 阶段参数已支持 `1a/1b/1a2/1b2/1a23/1b23` 等组合；但“同一进程内真正并行执行 L1B 与 L2”尚未内建。

## 3. 核心思路：双轨并行 + 同步栅栏

### 3.1 双轨模型

- **轨 A（文本智能轨）**
  - `L1A -> L2`
  - 产出：`current.comprehension`、`current.keep_mask`
- **轨 B（媒体几何轨）**
  - `L1A -> L1B -> VAD -> 候选接缝投机重编码`
  - 产出：静音索引与候选接缝重编码缓存（sidecar）

### 3.2 同步栅栏（Barrier）

- 在 L2 结束拿到 `keep_mask` 后，执行“最终切点规划”：
  - `keep_mask + annotations(t_start/t_end/gap_after) + padding/merge + VAD snap`
  - 生成最终 EDL（剪辑决策表）。
- L3 装配阶段优先消费轨 B 预编码缓存；未命中时按回退路径补编码。

## 4. 完整流程设计（目标形态）

### 4.1 阶段流程图（逻辑）

```text
L1A(ASR文本定稿)
  ├─> 轨A: L2(2a/2b...) ---------------------------┐
  └─> 轨B: L1B(强制对齐) -> VAD -> 候选接缝预编码 ----┤
                                                     │
                                   Barrier(等待 A/B) ┤
                                                     v
                          Final Planner(keep_mask+时间轴+snap)
                                                     |
                                                     v
                                   Assembler(命中缓存拼接/未命中回退)
```

### 4.2 分阶段说明

1. **L1A（前置）**
   - 输出可供 L2 立即消费的句面索引。
   - 同时产出 `audio_16k.wav`，为轨 B 提供统一音频源。
2. **轨 A：L2 执行**
   - 只依赖句面，不依赖对齐时间。
3. **轨 B：L1B + VAD + 候选接缝预编码**
   - L1B 回填句级时间后，可启动媒体几何预处理。
4. **Barrier**
   - 要求同时满足：
     - `keep_mask` 可用；
     - L1B 时间轴可用；
     - 轨 B 预处理状态可查询（可部分完成）。
5. **Final Planner**
   - 以最终参数确定真实切点集合。
6. **Assembler**
   - 对每个切点：先查缓存命中；未命中则即时补编码。

## 5. 关键设计细节与约束

### 5.1 “先预编码再 keep_mask”的本质

- 这是 **投机预计算**，不是精确预执行。
- 预编码集合必然是“最终切点集合”的超集或近似超集。
- 接受部分无效工作是方案前提；收益来自与 L2 的时间重叠。

### 5.2 切点不等于句缝

- 最终刀位由以下共同决定：
  - `keep_mask`
  - `pre_pad/post_pad`
  - `gap_after_cap`
  - `merge_short_intervals`
  - VAD snap（`silence_intervals + snap_radius`）
- 因此“L1B 句间边界”与“最终切点”通常不是一一对应。

### 5.3 候选接缝策略（可选）

| 策略 | 描述 | 计算量 | 命中率 | 适用阶段 |
|------|------|--------|--------|----------|
| 全句缝 | 所有相邻句边界均预编码 | 高 | 高 | POC/首版 |
| 启发式子集 | 仅对高概率切点预编码（长 gap、主题切换等） | 中 | 中 | 收敛期 |
| 网格窗口 | 时间网格切窗预编码，切点落窗即命中 | 可高可中 | 依参数 | 专项优化 |

建议：先做“全句缝 + 回退”，建立收益基线，再向启发式收敛。

### 5.4 回退机制必须存在

- 缓存未命中、缓存损坏、参数不一致、轨 B 未完成时：
  - 自动退回“即时局部重编码”路径；
  - 不允许因预处理失败阻断主流程成片。

### 5.5 并发写清单风险

- L2 与 L1B/轨B 并行时，禁止同时改写同一清单字段导致覆盖。
- 建议：
  - `timeline_manifest.json` 继续承载主契约字段；
  - 轨 B 状态与缓存索引写入 sidecar（独立文件）；
  - 使用“读-改-写锁”或单写协程保证原子性。

### 5.6 参数一致性与缓存键

- 缓存键至少包含：
  - `run_id`
  - 源媒体指纹（路径+mtime/size 或 hash）
  - `l1_contract` 关键字段（`segmentation_mode/sentence_endings/max_chars`）
  - 影响切点的执行参数（pad/merge/snap 相关）
  - 预编码器版本号
- 任一关键参数变化应判为 cache miss。

### 5.7 资源调度

- 若 L2 使用远端 LLM，轨 B 可充分占用本地 GPU，收益最大。
- 若 L2 与轨 B 同时争用本地 GPU（如本地推理），需提供配额或优先级策略。

## 6. 数据与产物约定

### 6.1 清单内（主契约）

- `annotations[]`：句面与时间轴（L1B 后完整）。
- `current.keep_mask[]`：L2 决策结果。
- `layer_status`：`l1a/l1b/l2/l3` 完成状态。

### 6.2 清单外（sidecar，建议）

- `silence_intervals` 索引文件。
- 候选接缝重编码缓存目录与索引文件。
- 轨 B 任务状态（进行中/完成/失败/版本信息）。

说明：sidecar 仅是性能加速层，不是语义真相源（SSOT）。

## 7. 风险与防护

| 风险 | 表现 | 防护策略 |
|------|------|----------|
| 预编码冗余过高 | GPU 时间上涨，收益反转 | 先做基线统计，设置预编码预算上限 |
| 清单并发覆盖 | 字段丢失或状态错乱 | sidecar 分离 + 单写控制 + 原子写 |
| 切点命中不足 | 最终仍大量即时编码 | 缓存键完善 + 策略迭代 + 保底回退 |
| 参数漂移 | 旧缓存误用 | 严格参数签名与版本钉 |
| 流程复杂度升高 | 运维与排障成本上升 | 加强日志分层与状态可观测性 |

## 8. 验收指标与实验方法

### 8.1 指标

- `T_end_to_end`：端到端耗时（核心指标）。
- `T_post_L2_to_output`：L2结束到出片耗时（关键路径指标）。
- `speculative_waste_ratio`：投机预编码中未被最终使用的比例。
- `cache_hit_ratio`：最终切点命中预编码缓存比例。
- 失败回退率：装配阶段触发即时编码的比例。

### 8.2 对照实验

- A 组：现有串行路径（L2 后再完整 L3）。
- B 组：双轨并行（L2 并行轨 B + 缓存装配）。
- 在同视频、同参数、同模型条件下对比上述指标，至少 5 轮取中位数。

---

## 9. 实施顺序建议

1. **P0：可观测性**
   - 为 L1A/L1B/L2/L3 以及轨 B 子任务补充分段计时与状态日志。
2. **P1：最小可用并行**
   - 先做 VAD sidecar（与 L2 并行），不改成片语义。
3. **P2：全句缝投机预编码 + 回退**
   - 建立真实收益基线。
4. **P3：候选接缝收敛**
   - 从全句缝收敛到启发式子集，降低冗余。
5. **P4：编排内建**
   - 将“双轨 + barrier”纳入统一 runner 生命周期管理。

## 10. 置信等级说明

### 10.1 已验证

- L1A 可直接供 L2（`tokens` 只读 `index/content`）。
- L3 当前必须依赖 L1B 后的 `t_start/t_end`。
- VAD 计算不依赖 keep_mask，可前置并行执行。

### 10.2 高置信

- 投机预编码可通过与 L2 并行降低墙钟时间（前提：本地资源可并行）。
- “全句缝预编码 -> 命中率高但冗余高；启发式预编码 -> 冗余低但命中率下降”是常见工程权衡。

### 10.3 待验证

- 在当前业务视频分布下，投机预编码是否稳定带来净收益（需实测）。
- 各候选接缝策略的最优参数区间（阈值、窗口宽度、预算上限）。
- 本地 GPU 与 L2 推理资源竞争场景下的最优调度策略。
