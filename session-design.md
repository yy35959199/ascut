  AutoSmartCut 智能层完整设计决策汇总

  > **归档说明（必读）**：本文为设计会话纪要，**不作为当前实现依据**。工程契约以 [doc/intelligence-layer2-mvp.md](intelligence-layer2-mvp.md) 与 [doc/AutoSmartCut-MVP.md](doc/AutoSmartCut-MVP.md) 为准。下文中的 `symbol_table` 持久化、`segments`/`keep_mask` 嵌套形态、2d 侧编译 `edl` 等与 **MVP 现行**（顶层 `keep_mask`、2a 程序消歧、执行层合成区间）可能不一致，请勿直接照抄进代码。

  一、周目与跨周目的信息流管理

  功能定义：
  一个周目 = 2a→2b→2c→2d 的完整循环。跨周目时创建新的
  TimelineManifest，将前一周目作为参考上下文，而非覆盖原数据。

  关键决策点：
  - 周目是独立的处理单元，每个周目有独立的 Manifest 实例
  - 新周目继承前一周目的反馈信息，但不修改前一周目的数据
  - 符合 Append-only 原则，保证数据可追溯性和重跑安全性
  - 跨周目反馈需要结构化，明确指向"理解问题"还是"决策问题"

  ---
  二、理解层（2a）的两轮处理流程

  功能定义：
  2a 分两轮处理，第一轮快速理解和识别，第二轮精化和确认。通过"理解先行、纠错后行
  "的策略，充分利用 LLM 的语义理解能力。

  Round 1 产出物（理解与候选识别）：
  - purpose_rough：粗略的整体视频主旨总结（一句话或简短段落）
  - segments_rough[]：初步的分块结构，每块包含标题、时间区间和简短总结
  -
  symbol_table_candidates[]：可能的易错词/误识词候选列表，含原始词、推测正确词和
   annotation index

  Round 2 产出物（确认与精化）：
  - purpose：精细化、校正好的整体主旨描述
  - segments[]：精细化的分块列表，每块标题和总结更准确、内容更丰富、时间范围确认
  - symbol_table[]：确认且经过筛选的易错词字典，仅保留高置信度条目
  - cleaned_annotations[]：基于确认的字典表对原始 ASR
  文本的虚拟纠正版本，形成人工可读且更准确的文本视图，不修改原始注释

  关键决策点：
  - LLM 的自然处理顺序是"先理解全文语义，再在理解基础上识别和纠错"，不同于人类的
  "先纠错再理解"
  - Round 1 在理解全文逻辑流的过程中顺便识别误识候选，认知负荷单一
  - Round 2 基于 Round 1 的理解和候选，做精确的确认和消歧
  - 易错词候选列表仅为中间产物，Round 2 产出后可丢弃
  - 符号表（字形映射）在纠错完成后价值有限，MVP 不做释义扩展，含释义版本放入
  after-mvp-todo

  ---
  三、分块与块总结的同轮处理

  功能定义：
  分块和块总结在 Round 1 同时完成，不需要分开为两个轮次。

  关键决策点：
  - LLM 的 attention
  机制支持"全局理解"，生成块总结时可同时查看该块内容和全文上下文
  - 不需要人类那样的"先读到块边再回头看"的过程
  - 分块和总结是同一个"理解"过程的两个输出，分开做反而低效（需要两次前向传播）
  - 一次前向传播完成所有任务，更高效

  ---
  四、文本消歧的虚拟修正方案

  功能定义：
  采用"符号表 + 虚拟消歧"方案，而非"原地修改"原始 ASR 文本。

  实现方式：
  - 原始 annotations[] 保持不变，完整保留 ASR 原始输出
  - 生成新的 cleaned_annotations[]，基于 Round 2 确认的符号表做虚拟消歧
  - 2b 决策层使用 cleaned_annotations[]，而非修改后的 annotations[]

  关键决策点：
  - 保留原始 ASR 文本的可追溯性，便于审计和问题排查
  - 避免 LLM 的"创意修改"（改变措辞、调整语序、过度修正）
  - 音轨和文本的对应关系不被破坏，虽然文本被虚拟修正，但时间戳和音轨仍然对应
  - 符号表本身是有价值的产物，记录了"什么被改成了什么"

  ---
  五、决策层（2b）的输入与输出

  功能定义：
  2b 基于消歧后的文本和理解层的产出，生成保留/删除决策。

  输入：
  - comprehension.purpose（精化的主旨）
  - comprehension.checklist[]（内容要点清单）
  - comprehension.cleaned_annotations[]（消歧后的文本）
  - annotations[]（句级语音，含 gap_after）

  输出：
  - segments[].keep_mask[]：每条 annotation 的保留决策（均为 true/false）
  - segments[].checklist_coverage[]：checklist 各项的覆盖情况

  关键决策点：
  - 不直接使用 symbol_table，符号表的知识已溶解进 cleaned_annotations
  - keep_mask 长度必须等于 annotations 长度，不得有缺失
  - 如果 LLM 返回的 keep_mask 长度不对，采用整体重跑（最多 3 次），不做局部修正

  ---
  六、审核层（2c）的 MVP 简化

  功能定义：
  MVP 阶段 2c 采用自动 pass 策略，不调用 LLM，为未来的真实审核预留接口。

  MVP 实现：
  - 代码层自动写入 verdict="pass"
  - 字段结构与完整版保持兼容

  关键决策点：
  - 这是有意的简化，不是架构缺陷
  - 完整版的 LLM 审核和循环机制在 MVP 后实现
  - 人工介入点固定在 2d，由用户在 CLI 中发现问题并反馈

  ---
  七、人工层（2d）的结构化反馈交互

  功能定义：
  2d 是唯一的人工介入点，通过 CLI 交互实现。采用 4
  个文本输入框引导用户按维度反馈，避免自由文本的模糊。

  四个输入框的具体内容：

  1. 主旨偏差反馈
    - 用户输入框：主旨是否有偏差？
    - 示例：应该强调"技术创新"而非"商业价值"
    - 触发处理：理解反馈，创建新周目重跑 2a
  2. 关键词识别错误反馈
    - 用户输入框：关键词或术语识别错了？
    - 示例：把"张伟"识别成了"章维"
    - 触发处理：纠错反馈，创建新周目重跑 2a（重跑模式，仅 1 次调用）
  3. 内容选择意见反馈
    - 用户输入框：内容选择上是否有方向性或具体意见？
    - 示例：第二段太冗长，应该删掉重复部分 / 缺少对 XXX 的讨论
    - 触发处理：理解反馈，创建新周目重跑 2a
  4. 剪辑时间节点意见反馈
    - 用户输入框：剪辑时间节点是否有意见（气口长短、紧凑度等）？
    - 示例：00:45-01:00 之间的停顿太长 / 这里的气口太紧凑
    - 触发处理：决策反馈，不创建新周目，直接在 2d 的 overrides 中处理

  其他交互操作：
  - [a] 确认通过：合并所有 overrides，编译 EDL，流转 Layer 3
  - [t] 切换片段：追加 override delta，刷新时长预览（暂时可选，不作 MVP 必需）
  - [r] 提交反馈：根据填写的输入框内容，触发对应的处理流程

  关键决策点：
  - 4 个输入框逼用户将模糊需求具体化，减少 LLM 解读负担
  - 代码层根据填写情况自动判断"应该修正什么"
  - 可视化片段勾选和时间边界拖拽作为未来可选交互，MVP 不必实现
  - 2d 的 CLI 界面应显示
  checklist_coverage[]，让用户一眼看到哪些要点被覆盖了、哪些没有

  ---
  八、跨周目反馈的层级映射

  功能定义：
  用户的 4 个输入框反馈需要映射到对应的处理层级，不同类型反馈走不同分支。

  映射规则：
  - 理解类反馈（输入框 1、2、3）→ 创建新周目，重跑 2a（重跑模式，仅 1 次调用）
  - 决策类反馈（输入框 4）→ 不创建新周目，直接在 2d 的 overrides 中处理
  - 理解反馈中，如果涉及"纠错"（输入框 2），2a 重跑时不重新消歧，继承前一周目的
  cleaned_annotations

  关键决策点：
  - 明确界面提示反馈将影响哪个层级和流程
  - 不必复杂跨层操作，分类映射至对应层级即可
  - 重跑 2a 时，根据反馈类型决定是否重新生成 symbol_table 和 cleaned_annotations

  ---
  九、Token 消耗监控与成本管理

  功能定义：
  MVP 阶段不做严格预算限制，仅实现 token 计数和成本估算的监控提示。

  监控内容：
  - 累计 input tokens（请求体内传给模型的 tokens）
  - 累计 output tokens（模型返回的 tokens）
  - 基于单价估算的累计消费金额

  实现方式：
  - 使用 DeepSeek SDK 的 Usage API 获取实际消耗
  - CLI 显示累计 token 数和估算费用，供用户了解
  - 写入日志便于分析和审计

  关键决策点：
  - MVP 不做上限限制，仅做监控提示
  - 缓存命中数据不易统计，MVP 阶段忽略
  - 后续调研 DeepSeek SDK 的具体 Usage 接口实现
  - 严格预算限制和循环终止策略放入 after-mvp-todo

  ---
  十、即时交互反馈"对话 Agent"功能规划

  功能定义：
  即时对话式反馈交互（用户说一句改一句的对话模式）不作为 MVP 目标功能。

  为什么不做：
  - 当前 LLM 多为语言理解，视频画面多模态能力不足
  - 多轮实时对话对模型计算和用户耐心都是挑战
  - 实际人机剪辑沟通中，反复改动的复杂性高，容易导致用户和系统都晕了
  - 性能和成本难以控制

  规划位置：
  - 放入 future.md 作为长期画饼规划
  - 未来版本考虑基于更强多模态模型、异步优化、渐进反馈流程实现

  ---
  十一、LLM 的"回头看"能力与前向传播

  功能定义：
  LLM 可以通过 attention 机制"回头看"输入的任何位置，不需要"顺序读"。

  技术原理：
  - Transformer 的 self-attention 允许任意位置之间的交互
  - 生成每个 token 时，LLM 可以查看输入的全部内容
  - 这不是"回头"，而是"全局查看"

  对设计的影响：
  - LLM 不需要人类那样的"先读到块边再回头看"的过程
  - 分块和总结可以在同一轮完成，无需分开
  - 一次前向传播可以完成多个任务，效率更高

  ---
  十二、2a 重跑模式的 Prompt 设计

  功能定义：
  当用户 [r] 反馈触发 2a 重跑时，需要特殊的 Prompt 设计来指导增量修正。

  关键决策点：
  - 将上一轮的 purpose 和 checklist 显式传入，告知 LLM 在已有理解的基础上调整
  - 明确告诉 LLM "你的任务是修正，不是重新理解"
  - 根据反馈类型决定是否重新生成 symbol_table 和 cleaned_annotations
  - 具体的 Prompt 模板设计待后续实现阶段细化

  ---
  十三、2b 的 Schema 验证与 Retry 策略

  功能定义：
  2b 生成的 keep_mask 必须通过严格的结构验证，确保长度和格式正确。

  验证规则：
  - len(keep_mask) == len(annotations)，不得有缺失 index
  - 每条 keep 必须为 true 或 false
  - 每条记录的 index 必须与对应 annotation 的 index 一致

  Retry 策略：
  - 检测到长度不对，直接 retry，最多 3 次
  - 如果 3 次都失败，报错退出
  - 不做局部修正，整体重跑更简单可靠

  关键决策点：
  - 在 2b 的 Prompt 中明确说明输出格式和约束
  - 代码层做严格的结构验证
  - 整体重跑而非局部修正，降低复杂度

  5. 关于 2d 人工层的反馈处理

  早期设计：
  [r] 自然语言反馈
    追加 feedback_text 到 human_feedback_history[]
    触发 2a 重跑
    2a 重跑将 feedback_text 作为触发原因注入 Prompt

  新决策的改变：
  - 反馈不再是"自由文本"，而是"4 个结构化输入框"
  - 不同类型的反馈触发不同的处理路径（理解反馈 → 2a，决策反馈 → 2d overrides）
  - 引入了"反馈映射规则"，明确指定哪些反馈走哪个分支
  - 推翻点：早期设计中所有反馈都通过 [r] 操作触发 2a 重跑，新决策分化了处理路径

  ---
  6. 关于符号表的价值和持久化

  早期设计：
  symbol_table[] 持久化保存在 comprehension 中
  作为"审计用，非下游输入"

  新决策的改变：
  - symbol_table 仍然持久化，但明确了"纯字形映射版本在纠错完成后价值有限"
  - 含释义版本（身份、观点等语义信息）才是真正有价值的，但 MVP 不做
  - 推翻点：早期设计中没有区分"字形映射"和"含释义"的符号表，新决策明确了这个区分

  ---
  7. 关于 2b 的输入

  2b 输入（从 manifest 读取）：
  - comprehension.purpose
  - comprehension.checklist[]
  - comprehension.cleaned_annotations[]（消歧后文本）
  - annotations[]（句级语音，含 gap_after）
  - 注意：comprehension.symbol_table 不在输入列表中

  Schema 验证与 Retry：keep_mask 长度或 index 不合规时整体重试（最多 3 次），不做局部修正。

  ---
  8. 关于 2c 的 MVP 简化

  早期设计：
  MVP 行为：代码层自动写入 verdict="pass"，不调用 LLM

  新决策的改变：
  - 保持不变，但新增了"2c auto-pass 的债务"的明确说明
  - 强调了 2d 的 CLI 界面应该显示 checklist_coverage，让用户发现漏洞
  - 推翻点：早期设计中没有明确说明 auto-pass 的后果和补救措施，新决策补充了这个

  ---
  9. 关于 2a 重跑时是否需要第三轮

  早期设计：
  没有明确讨论这个问题

  新决策的改变：
  - 明确决定：不需要第三轮 2a
  - 如果用户觉得理解完全错了，应该提供 [reset] 操作，清空 comprehension，从 2a
  冷启动重新开始
  - 推翻点：这是新增的决策，早期设计中没有涉及

  ---
  10. 关于 LLM 的处理顺序

  早期设计：
  没有从第一性原理深入讨论 LLM 的处理顺序

  新决策的改变：
  - 明确了 LLM 的自然顺序是"先理解后纠错"，而非人类的"先纠错后理解"
  - 这个认知改变了对两轮分工的理解
  - 推翻点：这是新增的深层理解，改变了对整个 2a 设计的认知基础

  # AutoSmartCut 智能层设计变更说明

  ## 1. 原有设计极简概述

  - 2a 理解层通过两轮 LLM 调用完成：
    - Round 1（bootstrap）：识别专有名词误识，输出 symbol_table
    - Round 2（消歧+理解）：应用 symbol_table，生成 cleaned_annotations 和 
  checklist
  - 符号表一次性产出，纠错完成后固化不变
  - 用户反馈统一通过 `[r]` 自然语言反馈触发 2a 重跑，不触碰固化字段
  - 2d 人工层反馈处理不区分类型，均触发重跑
  - 2b 决策层无严格输出验证策略
  - 2c 审核层 MVP 自动通过，无反馈闭环
  - LLM 处理流程未深究“理解先行还是纠错先行”

  ## 2. 主要变更点与理由

  | 变更点                           | 变更内容简介                             
                           | 变更理由（动机）                                   
                                                  |
  |---------------------------------|-------------------------------------------
  ------------------------|-----------------------------------------------------
  ---------------------------------------------|
  | 2a 分工细化                     | Round 1 
  输出新增分块及块总结；符号表分为候选和确认两阶段                | 保持 LLM 
  认知负荷合理，提升模块可调试性，提升分块理解和语义准确度                      
                   |
  | 纠错流程调整                   | 纠错与符号表确认合并到 Round 
  2，采用虚拟消歧而非文本原地修改             | 
  保留原文溯源，避免语义漂移和创意改写，保证音轨与文本一致性                    
                           |
  | 反馈处理分类                   | 
  反馈分为理解类和决策类，分别输入对应层（理解反馈触发 2a 重跑，决策反馈在 
  overrides） | 减少不必要重跑，精准修改对应模块，提升反馈效率与用户体验

                   |
  | 2b 输出验证与重试机制           | 明确了 keep_mask

  保证输出格式正确，避免模型偶尔输出结构错误带来的下游故障
                          |
  | LLM 处理流程理解                | 明确 LLM 先理解语义再纠错，与人类顺序相反

  简单稳定，提升人工审阅效率，预留未来完善空间
