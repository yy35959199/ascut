● 你的项目 AutoSmartCut 是一个基于时间轴的多层次语义剪辑系统，核心围绕一个名为
  TimelineManifest 的时间轴语义清单。它通过“三层管道”逐步处理原始视频媒体：

  1. 识别层（Layer 1）：
    - 输入视频，提取音频，进行语音识别（ASR）产生带时间戳的语音标注。
    - 标注内容包括语音段落、静音和场景切换等。
    - 输出时间有序的注释列表。
  2. 智能层（Layer 2）：
    - 使用大型语言模型（LLM）进行多轮推理，分阶段完成：
        - 2a 理解子阶段：生成内容摘要、符号表、校正标注等。
      - 2b 决策子阶段：基于校正结果生成剪辑片段建议及覆盖报告。
      - 2c 审核子阶段：自动审核通过或失败，提示是否需要人工介入。
      - 2d 人工子阶段：人工参与复核、反馈和编辑最终剪辑决策。
    - 2a到2d形成闭环多轮迭代。
    - 输出带有保留或者剪辑决策的片段列表及审核报告。
  3. 执行层（Layer 3）：
    - 基于第2层的剪辑决策执行视频剪辑操作。
    - 使用 smartcut 库对 GOP 级别的视频进行高精度剪切和复编码（HEVC hybrid
  recode）。
    - 生成最终剪辑后的视频文件。

  设计原则：
  - 统一接口：所有节点处理输入清单，追加生成输出清单，实现流水线异步并行。
  - 只增不改：数据结构无破坏式修改，只追加信息，保证历史数据回溯。
  - 开放扩展：开放元信息字段，方便后续自然向前兼容的功能扩展。
  - 多轮循环迭代和人机协作，保证剪辑准确性和灵活性。
  - 支持断点续跑、版本控制和审计跟踪，保证可追溯与可靠运行。

  MVP落地文档细化了核心模块的具体实现细节，包括数据模型、接口定义、演示脚本和技
  术决策，验证了主流程的端到端通路。MVP阶段实现：
  - Layer 1: 使用 Qwen3-ASR + ForcedAligner 进行语音转录和字级时间戳生成。
  - Layer 2: 固定两轮LLM推理和1轮独立调用决策，自动passthrough自动通过机制，人工
  阶段通过CLI交互处理反馈和修正。
  - Layer 3: 通过 smartcut 库精准 GOP 切割，执行剪辑生成最终视频。
  - 整体无插件系统，遵守只增不改，不支持多轮跨层循环，确保核心功能稳健。

  项目结构有单独仓库
  AutoSmartCut，内含包括核心模块：识别（perception.py）、智能（intelligence.py）
  、执行（execution.py）和调度控制（runner.py）。配合Demo脚本逐步验证各个阶段。

  如果你需要，我可以帮你具体分析某个模块代码，或者帮你梳理执行流程。

  ## 1. 设计原则                                                                
                                                                                
  **P1：manifest 是唯一通信介质**                                               
                                                                                
  步骤函数之间不直接传参。每个步骤从 manifest                                   
  读自己需要的字段，写回自己产出的字段。调度器不手动串联中间变量，manifest      
  的当前状态即是完整上下文。                                                    
                                                                                
  **P2：步骤函数各自独立，不共享基类**                                          
                                                                                
  四个子阶段（2a/2b/2c/2d）共享同一外层契约 `f(manifest_state) →                
  manifest_mutations`，但内部逻辑差异大：prompt 构造完全不同，2d 根本不调       
  LLM。共享的机械动作（HTTP 调用、JSON 解析、重试、token 计数）提取为           
  `llm_call()` 工具函数，不用基类。                                             
                                                                                
  **P3：状态机控制流程，显式优于表驱动**                                        
                                                                                
  控制流转（2a→2b→2c→2d 及各种回溯路径）通过 ~40 行显式 `if/elif`               
  状态机实现，任何人看一眼即知流程走向。状态转移只在四个外部边界上发生，不在 2a 
   的内部轮次间发生。                                                           
                                                                                
  **P4：Append-only 保证重跑安全**                                              
                                                                                
  重跑时，固化字段（`symbol_table`、`cleaned_annotations`）不被覆盖，只有可变字 
  段（`purpose`、`checklist`）被更新。Append-only                               
  原则天然防止重跑时引入不一致。                                                
                                                                                
  ---                                                                           
                                                                                
  ## 2. 整体结构                                                                
                                                                                
  ### 2.1 三个构件                                                              
                                                                                
  ```                                                                           
  intelligence.py                                                               
  ├── llm_call()           ← 共享传输层                                         
  ├── step_2a()            ← 理解层（含冷启动/重跑两种内部模式）                
  ├── step_2b()            ← 决策层                                             
  ├── step_2c()            ← 审核层（MVP: auto-pass）                           
  ├── step_2d()            ← 人工层（CLI 交互，不调 LLM）                       
  └── run_intelligence()   ← 状态机主循环                                       
  ```                                                                           
                                                                                
  **`llm_call()`** — 共享传输层                                                 
                                                                                
  处理所有与语义无关的机械动作：                                                
                                                                                
  ```python                                                                     
  @dataclass                                                                    
  class LLMResponse:                                                            
      parsed: dict        # 解析后的 JSON                                       
      tokens_in: int                                                            
      tokens_out: int                                                           
      raw_content: str    # 原始响应，用于审计                                  
                                                                                
  def llm_call(                                                                 
      client: OpenAI,                                                           
      model: str,                                                               
      messages: list[dict],                                                     
      *,                                                                        
      max_retries: int = 3,                                                     
      temperature: float = 0.3,                                                 
  ) -> LLMResponse:                                                             
      ...                                                                       
  ```                                                                           
                                                                                
  **`StepOutput`** — 步骤函数的统一返回类型                                     
                                                                                
  ```python                                                                     
  @dataclass                                                                    
  class StepOutput:                                                             
      mutations: dict        # 要写回 manifest 的字段                           
      tokens_used: int                                                          
      signal: str | None     # 控制信号，仅 2c/2d 产出                          
                             # "2a" | "2b" | "2d" | "done"                      
  ```                                                                           
                                                                                
  ### 2.2 完整状态机流转图                                                      
                                                                                
  ```mermaid                                                                    
  flowchart TD                                                                  
      entry([入口]) --> guard_init[初始化守卫\nouter=0 inner=0 tokens=0]        
      guard_init --> stage_2a                                                   
                                                                                
      subgraph stage2a [2a 理解层]                                              
          stage_2a{cleaned_annotations\n已存在?}                                
          cold[冷启动模式\nR1+R2 共2次调用]                                     
          rerun[重跑模式\n1次调用]                                              
          stage_2a -- 否 --> cold                                               
          stage_2a -- 是 --> rerun                                              
      end                                                                       
                                                                                
      cold --> write_2a[写入: purpose + checklist\ncleaned_annotations +        
  symbol_table]                                                                 
      rerun --> write_2a_r[写入: purpose + checklist\n不动                      
  symbol_table/cleaned_annotations]                                             
      write_2a --> stage_2b                                                     
      write_2a_r --> stage_2b                                                   
                                                                                
      subgraph stage2b [2b 决策层]                                              
          do_2b[1次 LLM 调用]                                                   
      end                                                                       
      stage_2b --> do_2b                                                        
      do_2b --> write_2b[写入: keep_mask\nchecklist_coverage]                   
      write_2b --> guard[终止守卫]                                              
                                                                                
      guard --> stage_2c                                                        
                                                                                
      subgraph guard_box [终止守卫 代码层判断]                                  
          guard{"token_spent >= budget\nOR rounds >= max\nOR coverage >= 0.9"}  
      end                                                                       
                                                                                
      guard -- 强制pass --> stage_2c                                            
                                                                                
      subgraph stage2c [2c 审核层]                                              
          do_2c[完整版: LLM审核\nMVP: auto-pass]                                
      end                                                                       
      stage_2c --> do_2c                                                        
      do_2c --> verdict{verdict}                                                
                                                                                
      verdict -- fix_checklist --> inc_outer[outer_rounds++]                    
      inc_outer --> stage_2a                                                    
                                                                                
      verdict -- fix_decision --> inc_inner[inner_rounds++]                     
      inc_inner --> stage_2b                                                    
                                                                                
      verdict -- pass --> stage_2d                                              
                                                                                
      subgraph stage2d [2d 人工层]                                              
          human_op{操作}                                                        
      end                                                                       
      stage_2d --> human_op                                                     
                                                                                
      human_op -- "[a] 确认" --> merge[merge keep_mask\n+ 所有 overrides]       
      merge --> edl[编译 edl[]]                                                 
      edl --> done([layer_completed=2\n流转 Layer 3])                           
                                                                                
      human_op -- "[t] 切换" --> overrides[追加 overrides delta\n刷新时长预览]  
      overrides --> stage_2d                                                    
                                                                                
      human_op -- "[r] 反馈" --> feedback[追加 feedback_text\nsignal = '2a']    
      feedback --> stage_2a                                                     
  ```                                                                           
                                                                                
  ### 2.3 MVP 退化形态                                                          
                                                                                
  MVP 中循环禁用（`max_inner=0, max_outer=0`），2c                              
  auto-pass，状态机退化为线性序列：                                             
                                                                                
  ```mermaid                                                                    
  flowchart LR                                                                  
      entry([入口]) --> A                                                       
      A["2a 冷启动\nR1+R2"] --> B                                               
      B["2b 决策\n1次调用"] --> C                                               
      C["2c auto-pass\n写入 verdict=pass"] --> D                                
      D["2d 人工审阅\nCLI"]                                                     
                                                                                
      D -- "[a] 确认" --> edl([edl[] 写入\n流转 Layer 3])                       
      D -- "[t] 切换" --> D                                                     
      D -- "[r] 反馈" --> A2["2a 重跑\n1次调用\n不重新消歧"]                    
      A2 --> B                                                                  
  ```

  ## 3. 2a 理解层                                                               
                                                                                
  ### 3.1 对外契约                                                              
                                                                                
  **输入：**                                                                    
  - `manifest.annotations[]`（Layer 1 产出的原始标注，含 ASR 误识）             
  - `manifest.goal`（用户指定的分析目标）                                       
                                                                                
  **输出（写入 `manifest.comprehension`）：**                                   
  - `purpose`：视频内容的核心目标与关键信息点                                   
  - `checklist[]`：结构化内容要点，含 must/optional 优先级                      
  - `cleaned_annotations[]`：消歧后的标注文本（原始标注保持不变，Append-only）  
                                                                                
  **审计字段（不向下游传递）：**                                                
  - `symbol_table[]`：专有名词的误识形式与正确形式对照表，持久化供审计          
                                                                                
  > **关键约束：** 2b/2c 的输入不包含 `symbol_table`。符号表的知识在 2a Round 2 
   中已溶解进                                                                   
  `cleaned_annotations`，下游消费的是消歧后的文本，不需要知道原始映射。         
                                                                                
  ### 3.2 comprehension 字段稳定性分类                                          
                                                                                
  | 字段 | 稳定性 | 来源 | 重跑行为 |                                           
  |------|--------|------|---------|                                            
  | `symbol_table` | **固化**（Round 0 后不变）| 从不可变的 ASR 原文推导 |      
  不重算 |                                                                      
  | `cleaned_annotations` | **固化**（Round 0 后不变）| 从 ASR 原文 +           
  symbol_table 推导 | 不重算 |                                                  
  | `purpose` | **可变** | 对内容的解读角度，受反馈/约束影响 | 每轮可更新 |     
  | `checklist` | **可变** | 同上 | 每轮可更新 |                                
                                                                                
  稳定性的根本依据：`symbol_table` 和 `cleaned_annotations` 的输入是不可变的    
  ASR 原文，只要 ASR                                                            
  输出不变它们就不会变。重跑消歧的边际收益为零，且随机温度可能引入噪声。        
                                                                                
  ### 3.3 冷启动模式（Round 0）                                                 
                                                                                
  **触发条件：** `manifest.comprehension` 为空或 `cleaned_annotations` 为空。   
                                                                                
  **执行：两次 LLM 调用**                                                       
                                                                                
  ```                                                                           
  Round 1（bootstrap）                                                          
    输入: annotations 原始文本 + goal                                           
    输出: { purpose_rough, symbol_table[] }                                     
    目的: 先聚焦识别专有名词误识，认知负荷单一                                  
                                                                                
  Round 2（消歧+理解）                                                          
    输入: annotations 原始文本 + goal                                           
          + purpose_rough（Round 1 产出）                                       
          + symbol_table[]（Round 1 产出）                                      
    输出: { purpose, checklist[], cleaned_annotations[] }                       
    目的: 拿着已有结论做查表确认式消歧，同时精化理解                            
  ```                                                                           
                                                                                
  **两轮分离的理由：** 若一轮完成，LLM 必须同时做"发现系统性误识"和"根据误识做  
  消歧"这两件事，认知负荷过大，一致性难以保证。两轮分工明确，Round 1            
  识别，Round 2 应用，每轮任务单一。                                            
                                                                                
  ### 3.4 重跑模式（Round 1+）                                                  
                                                                                
  **触发条件：** `cleaned_annotations` 已存在（重跑 2a 时）。                   
                                                                                
  **执行：一次 LLM 调用**                                                       
                                                                                
  ```                                                                           
  输入:                                                                         
    cleaned_annotations[]   ← 已有，不重新生成                                  
    goal                    ← 不变                                              
    上一轮 purpose          ← 作为增量修正的起点                                
    上一轮 checklist[]      ← 作为增量修正的起点                                
    触发原因（二选一）:                                                         
      completeness_issues[] ← 2c 发现检查清单遗漏维度（外循环）                 
      feedback_text         ← 用户自然语言反馈（2d [r] 操作）                   
                                                                                
  输出:                                                                         
    { purpose（更新）, checklist[]（更新）}                                     
    不产出 symbol_table / cleaned_annotations（不触碰固化字段）                 
  ```                                                                           
                                                                                
  **增量修正的 Prompt 策略：** 将上一轮的 `purpose` 和 `checklist`              
  显式传入，告知 LLM 在已有理解的基础上按触发原因调整，而不是从零重建。         
                                                                                
  ### 3.5 符号表（Symbol Table）的定位                                          
                                                                                
  **为什么需要：**                                                              
                                                                                
  ASR 误识具有**系统性**：同一个专有名词在同一视频中往往被一致误识成同一个错误  
  形式（相同说话人、相同声学环境、模型对特定音节的固定偏好）。符号表把 Round 2  
  的任务从"开放式识别哪里有错"降维为"按表批量替换+语境确认"，保证纠正一致性。   
                                                                                
  **收录范围：**                                                                
                                                                                
  | 类别 | 收录标准 |                                                           
  |------|---------|                                                            
  | 人名 | 出现 ≥2 次，且姓名组合不常见 |                                       
  | 专有术语 | 领域词汇，ASR 容易拆解为日常词组 |                               
  | 产品/项目名 | 中英文混合场景误识高发区 |                                    
  | 机构/地名 | 低频专有名词 |                                                  
                                                                                
  不收录：单次随机口误、日常词汇偶发错误、句法层面的错误（符号表只解决字级误识  
  ）。                                                                          
                                                                                
  MVP 预期条目数：**5–20 条**（取决于视频专业程度）。                           
                                                                                
  **生命周期：**                                                                
                                                                                
  ```                                                                           
  ASR 原文（含误识）                                                            
      │                                                                         
      │ [2a Round 1]                                                            
      ├──────────────→ symbol_table     ← 在此建立，R1 的工作台                 
      │                                                                         
      │ [2a Round 2]                                                            
      ├──────────────→ cleaned_annotations  ← 知识溶解于此                      
      │               （symbol_table 的使命完成）                               
      │                                                                         
      │ [持久化]                                                                
      └──────────────→ comprehension.symbol_table  ← 审计用，非下游输入         
  ```                                                                           
                                                                                
  **关于 RAG：** MVP 及长视频场景均不需要 RAG。5–10 分钟视频的总文本量约        
  3000–6000 字，远在 DeepSeek 128K                                              
  上下文窗口内。长视频场景的正确方案是**符号表作全局头部注入 +                  
  滑窗分块**：每个分块的 Prompt 以 `[全局头: purpose + checklist +              
  symbol_table]` 开头，后接本块的 annotations 切片。符号表约 600 字，相比 RAG   
  管线简单且无信息损失。                                                        
                                                                                
  ---                                                                           
                                                                                
  ## 4. 2b 决策层                                                               
                                                                                
  **输入（从 manifest 读取）：**                                                
  - `comprehension.purpose`                                                     
  - `comprehension.checklist[]`                                                 
  - `comprehension.cleaned_annotations[]`（消歧后文本）                         
  - `annotations[]` 中的 silence 条目（用于确定 keep=null 的位置）              
                                                                                
  > 注意：`comprehension.symbol_table` **不在输入列表中**。                     
                                                                                
  **输出（写入 manifest）：**                                                   
  - `segments[].keep_mask[]`：每条 annotation 的保留决策                        
  - `segments[].checklist_coverage[]`：checklist 各项的覆盖情况                 
                                                                                
  **keep_mask 格式约束：**                                                      
  - `len(keep_mask) == len(JSON2.tokens)`，不得有缺失 index                     
  - speech 条目：`keep=true/false`（LLM 决策）                                  
  - silence 条目：`keep=null`（不由 LLM 决策，由规则推导：两侧 speech           
  均保留则静音保留）                                                            
                                                                                
  ---                                                                           
                                                                                
  ## 5. 2c 审核层                                                               
                                                                                
  **完整版行为：**                                                              
  - 输入：`purpose + checklist[] + 当前 keep_mask`                              
  - LLM 逐项核查每条 must 项是否被实质覆盖                                      
  - 输出 `verdict ∈ {pass, fix_decision, fix_checklist}`                        
                                                                                
  | verdict | 含义 | 后续动作 |                                                 
  |---------|------|---------|                                                  
  | `pass` | must 项全覆盖，无明显噪声 | 流转 2d |                              
  | `fix_decision` | must 项未覆盖，但清单本身完整 | 回到 2b（内循环），携带    
  `coverage_issues[]` |                                                         
  | `fix_checklist` | 清单遗漏重要维度 | 回到 2a（外循环），携带                
  `completeness_issues[]` |                                                     
                                                                                
  **MVP 行为：** 代码层自动写入 `verdict="pass"`，不调用                        
  LLM。字段结构与完整版保持兼容，未来启用真实审核只需修改生成逻辑，不改         
  schema。                                                                      
                                                                                
  ---                                                                           
                                                                                
  ## 6. 2d 人工层                                                               
                                                                                
  2d 是整条管道唯一的人工介入点，通过 CLI 交互实现，不调用 LLM。                
                                                                                
  | 操作 | 行为 | LLM 调用 |                                                    
  |------|------|---------|                                                     
  | `[a]` 确认通过 | merge(keep_mask + 所有历史 overrides) → 编译 `edl[]` | 否  
  |                                                                             
  | `[t N]` 切换 keep/cut | 追加 `overrides`                                    
  delta，刷新时长预览，等待下一步操作 | 否 |                                    
  | `[r]` 自然语言反馈 | 追加 `feedback_text` 到                                
  `human_feedback_history[]`，触发 2a 重跑 | 是（重跑 2a）|                     
                                                                                
  **overrides 的 delta 模式：**                                                 
                                                                                
  `[t]` 操作不修改 `segments[].keep_mask` 原地值，而是向当前                    
  `HumanFeedbackRound.overrides[]` 追加 delta 记录 `{"index": int, "keep":      
  bool}`。最终有效决策：                                                        
                                                                                
  ```                                                                           
  effective_keep[i] = 最后一条覆盖 index i 的 override（若有）                  
                      否则 keep_mask[i]                                         
  ```                                                                           
                                                                                
  `[a]` 确认时一次性合并所有 overrides 编译为 `edl[]`，这是 `edl[]`             
  的唯一写入点。                                                                
                                                                                
  **`[r]` 触发重跑的行为：**                                                    
  - 追加一条 `HumanFeedbackRound(verdict="feedback", feedback=<用户输入>)`      
  - 状态机 signal 返回 `"2a"`，进入 2a **重跑模式**（不冷启动，不重新消歧）     
  - 2a 重跑将 `feedback_text` 作为触发原因注入 Prompt

  ---
  1. 理解层重跑时发现理解问题怎么办？

  问题本质： 用户 [r] 反馈说"你理解错了"，2a
  重跑（1次调用）能否修正？还是需要第三轮？

  逻辑分析：

  2a 重跑的输入是：
  - cleaned_annotations（已固化，不变）
  - 上一轮的 purpose 和 checklist
  - 用户的 feedback_text

  如果用户反馈是"你理解错了"，那么：
  - 情况A：理解错误源于 ASR 误识 → cleaned_annotations 本身有问题 → 重跑 1
  次调用无法修正（因为输入没变）
  - 情况B：理解错误源于 LLM 的解读角度偏差 → 用户反馈可以纠正 → 重跑 1
  次调用足够

  依据：

  符号表和消歧标注的设计假设是"ASR 误识具有系统性，可以在 Round 1
  一次性识别"。但实际上：
  - Round 1 可能漏掉某些误识（低频专有名词、首次出现的术语）
  - Round 2 的消歧可能不完整（LLM 没有严格按符号表替换）

  如果用户反馈指出了这类问题，重跑 1 次调用无法修正，因为 cleaned_annotations
  已固化。

  结论：

  不应该跑第三轮 2a。 原因：

  1. 架构一致性：固化字段的设计原则是"输入不变则输出不变"。如果允许重跑时修改固
  化字段，Append-only 原则就被破坏了。
  2. 问题根源：如果 cleaned_annotations 有问题，说明 Round 1
  的符号表不完整。正确的修复路径是：
    - 用户在 2d 发现问题后，手动在 CLI 中标注哪些片段有误识
    - 或者，提供一个 [reset] 操作，清空 comprehension，从 2a
  冷启动重新开始（相当于重新跑 Round 1+2）
  3. MVP 的简化：MVP 阶段不处理这种情况。如果用户发现消歧有问题，只能：
    - 接受现状，手动在 2d 调整 keep/cut
    - 或者，退出程序，删除检查点，重新跑一遍

  建议：

  在 2d 的 CLI 中加一个 [reset] 操作：
  [reset]  清空理解层结果，从 2a 冷启动重新开始（慎用）

  这样用户有一个"重来"的选项，但不破坏架构的一致性。

  ---
  2. 2b 长度不对时 LLM 该怎么办？

  问题本质： keep_mask 长度必须等于 annotations 长度。如果 LLM
  返回的长度不对，是让 LLM 局部修正还是整体重跑？

  逻辑分析：

  长度不对的原因可能是：
  - LLM 漏掉了某些 annotation（比如跳过了某些 silence 条目）
  - LLM 多生成了某些 index（比如重复了某个 index）
  - LLM 理解错了任务，返回了完全不同的结构

  依据：

  如果让 LLM "局部修正"，需要：
  1. 代码层检测出哪些 index 缺失
  2. 构造一个新的 Prompt，告诉 LLM "你漏掉了 index X, Y, Z，请补充"
  3. LLM 返回补充的部分，代码层合并

  这个方案的问题：
  - 增加了一次额外的 LLM 调用
  - 合并逻辑复杂，容易出错
  - 如果 LLM 第二次还是返回错误的结构，怎么办？

  结论：

  整体重跑，不做局部修正。 原因：

  1. 简单可靠：检测到长度不对，直接 retry，最多 3 次。如果 3
  次都失败，报错退出。
  2. Prompt 优化：在 2b 的 Prompt 中明确说明：
  你必须为每一条 annotation 生成一个决策，不能遗漏。
  输出的 keep_mask 数组长度必须等于 {len(annotations)}。
  3. Schema 验证：在代码层做严格的结构验证：
  def validate_keep_mask(keep_mask: list, annotations: list) -> bool:
      if len(keep_mask) != len(annotations):
          return False
      for i, entry in enumerate(keep_mask):
          if entry['index'] != annotations[i].index:
              return False
          if annotations[i].type == 'silence' and entry['keep'] is not None:
              return False
          if annotations[i].type == 'speech' and entry['keep'] not in [True,
  False]:
              return False
      return True
  4. Retry 策略：
  for attempt in range(3):
      response = llm_call(...)
      if validate_keep_mask(response['keep_mask'], annotations):
          break
      else:
          # 记录错误，重试
          log.warning(f"keep_mask validation failed, attempt {attempt+1}/3")
  else:
      raise ValueError("2b failed after 3 attempts")

  建议：

  在 intelligence-design.md 中补充一节"2b 的 Schema 验证与 Retry 策略"。

  3. 2c 在 demo2 第一轮先不做

  理解： Demo2 的目标是验证 2a→2b→2d 的完整链路，2c auto-pass
  先不实现，等链路跑通后再加。

  结论：

  同意。 这是合理的增量开发策略。

  建议的实现顺序：

  1. Demo2 第一轮：

    - 实现 step_2b()（1 次 LLM 调用）

  2. Demo2 第二轮：
    - 加入 step_2c() auto-pass（代码层写入 verdict="pass"）
    - 加入 step_2d() 的 [t] 切换和 [r] 反馈

  4. 符号表的简单规则

  问题： MVP 阶段符号表的收录标准应该是什么？

  逻辑分析：

  符号表的目的是"识别 ASR 的系统性误识"。MVP 阶段不需要完美，只需要能工作。

  简单规则（给 LLM 的 Prompt）：

  请识别视频中可能被 ASR 误识的专有名词，包括：
  1. 人名（中文姓名、外国人名的音译）
  2. 专业术语（技术词汇、行业黑话）
  3. 产品/项目名称（尤其是中英文混合的）
  4. 机构/地名（不常见的组织名、地点名）

  收录标准：
  - 在转写文本中出现 ≥2 次
  - 你认为 ASR 可能将其误识为其他同音词
  - 不收录日常词汇和常见表达

  输出格式：
  {
    "symbol_table": [
      {
        "term": "张伟",           // 正确形式
        "raw_form": "章维",       // ASR 可能的误识形式
        "category": "person",     // person | term | entity | other
        "first_occurrence": 3     // 首次出现的 annotation index
      }
    ]
  }

  如果没有发现明显的误识，返回空数组。

  依据：

  - "出现 ≥2 次"：过滤掉偶发的随机口误
  - "你认为 ASR 可能误识"：把判断权交给 LLM，不需要代码层做复杂的规则
  - "不收录日常词汇"：避免符号表过大

  结论：

  这个规则足够简单，LLM 可以理解，MVP 阶段够用。

  后续优化方向（记录到 after-mvp-todo.md）：

  - 符号表的自动验证：检查 raw_form 是否真的在原始 annotations 中出现
  - 符号表的置信度：给每条记录加一个 confidence 字段，表示 LLM
  对这个误识的确信程度
  - 符号表的人工编辑：允许用户在 2d 手动添加/删除符号表条目

  6. Token 预算建议

  问题： MVP 中固定 3 次 LLM 调用，但用户可以无限 [r] 反馈，成本可能失控。

  逻辑分析：

  Token 预算的目的是：
  1. 控制成本（避免单个视频消耗过多 token）
  2. 防止无限循环（用户反复 [r] 反馈但问题没有改善）

  依据：

  DeepSeek 的成本极低（约 $0.14 / 1M input tokens，$0.28 / 1M output
  tokens），20 分钟视频的单次处理成本约：
  - 2a Round 1: ~20K input + ~2K output = $0.003
  - 2a Round 2: ~20K input + ~5K output = $0.004
  - 2b: ~20K input + ~3K output = $0.003
  - 总计：~$0.01 / 次

  如果用户 [r] 反馈 10 次，总成本约 $0.10，仍然可以接受。

  结论：

  MVP 阶段不需要严格的 token 预算，但需要一个软上限。 建议：

  1. 反馈次数上限：最多允许 10 次 [r] 反馈（每次触发 1 次 2a 重跑）
  MAX_FEEDBACK_ROUNDS = 10

  if len(manifest.human_feedback_history) >= MAX_FEEDBACK_ROUNDS:
      print(f"已达到最大反馈次数 ({MAX_FEEDBACK_ROUNDS})，无法继续 [r] 操作")
      print("请使用 [a] 确认当前结果，或 [reset] 重新开始")
  2. Token 消耗提示：在每次 LLM 调用后，显示累计 token 消耗
  [2a Round 1 完成] tokens: 22,341 input + 1,876 output (累计: $0.003)
  [2a Round 2 完成] tokens: 21,098 input + 4,523 output (累计: $0.007)
  [2b 完成] tokens: 19,765 input + 2,987 output (累计: $0.010)
  3. 成本预警：如果累计 token 消耗超过某个阈值（比如 $0.50），给出警告
  ⚠️  警告：本次处理的累计成本已超过 $0.50，建议尽快完成或重新开始

  后续优化方向（记录到 after-mvp-todo.md）：

  ## Token 预算与成本控制

  **完整版方案：**
  1. 按视频时长动态分配 token 预算：
     - 5–10 分钟：预算 100K tokens
     - 10–30 分钟：预算 200K tokens
     - 30–60 分钟：预算 500K tokens
  2. 循环终止条件：
     - `token_spent >= token_budget`
     - `rounds >= max_rounds`
     - `coverage >= early_stop_threshold`（默认 0.9）
  3. 预算耗尽时的行为：
     - 强制 2c 返回 `verdict="pass"`
     - 流转到 2d，由人工兜底

  Q1：2a 一轮内同时做 4 项任务，LLM 能否做好？

  从第一性原理分析：

  LLM 的 Transformer 架构确实基于注意力机制，但"注意力分散"这个直觉在 LLM
  上的表现不同于人类：

  人类的注意力分散：
  - 同时做 4 件事 → 每件事的质量都下降
  - 原因：人类的工作记忆容量有限（7±2 个 chunk）

  LLM 的"注意力"：
  - 不是"同时处理"，而是"按 token 序列顺序处理"
  - 一个 Prompt 中的 4 项任务，LLM 会按照 Prompt 的结构化程度来决定是否"混淆"

  关键变量：Prompt 的结构化程度

  低结构化 Prompt：
  "请分析这个视频的主旨、分块、纠错和字典表"
  → LLM 可能混淆，某些任务做得不好

  高结构化 Prompt：
  "
  ## 任务 1：主旨识别
  请用一句话总结视频的核心主旨。
  输出格式：{"purpose": "..."}

  ## 任务 2：分块识别
  请按叙事节奏将内容分为 3–5 块，每块用一句话总结。
  输出格式：{"segments": [{"title": "...", "summary": "..."}]}

  ## 任务 3：误识识别
  请列出可能被 ASR 误识的专有名词（出现 ≥2 次）。
  输出格式：{"candidates": [{"term": "...", "raw_form": "..."}]}

  ## 任务 4：字典表
  请从候选误识中确认哪些是真正的误识。
  输出格式：{"symbol_table": [...]}
  "
  → LLM 会按顺序完成每个任务，质量更高

  我的结论：

  LLM 可以在一轮内做好 4 项任务，但前提是 Prompt 足够结构化。 关键是：

  1. 显式分解任务：不是"请做 4 件事"，而是"任务 1 是 X，任务 2 是 Y，..."
  2. 明确输出格式：每个任务都指定 JSON schema，LLM 按 schema 输出
  3. 任务间的依赖关系：如果任务 4 依赖任务 3 的输出，要明确说明

  但是， 有一个更深层的问题：这 4 项任务是否应该在同一轮完成？

  从认知负荷的角度，我认为应该分两轮：

  - Round 1：识别与假设（任务 1、2、3）
    - 目标：快速识别"有什么"
    - 输出：粗粒度的主旨、分块、误识候选
  - Round 2：确认与应用（任务 3 的确认、任务 4）
    - 目标：精化理解、确认误识、生成消歧文本
    - 输入：Round 1 的输出
    - 输出：精化的主旨、分块、确认的 symbol_table、cleaned_annotations

  理由：

  1. 信息流清晰：Round 1 的输出是 Round 2 的输入，形成显式的"粗→细"流程
  2. LLM 的认知负荷更低：每轮的任务更单一，质量更高
  3. 可调试性更好：如果 Round 2 的结果不好，可以看是 Round 1 的假设错了，还是
  Round 2 的确认错了

  所以我的答案是： 不是"一轮内 4 项任务"，而是"两轮内 4 项任务，每轮 2 项"。

  Q1：文章分块和块总结，应该分开还是合并？

  从信息流角度分析：

  分块和总结的依赖关系��：
  - 分块 需要理解整体结构（需要读完全文）
  - 块总结 需要已有分块结果（依赖分块的输出）

  从 LLM 的认知负荷角度：

  如果在同一轮内同时做分块和总结：
  LLM 需要：
  1. 理解全文逻辑流
  2. 识别分块边界
  3. 为每块生成总结

  这三个任务是串行依赖的，不是并行的。LLM 必须先完成 1 和 2，才能做 3。

  我的结论：应该合并到同一部分。

  理由：
  1. 它们是同一个"理解"过程的两个输出：LLM 理解全文时，自然会识别结构和要点
  2. 分开做反而低效：如果分开，第二轮需要重新读全文来生成总结，浪费计算
  3. 输出结构可以统一：segments[] 中每条既包含分块信息，也包含总结

  建议的数据结构：

  @dataclass
  class Segment:
      index: int
      title: str              # 分块标题
      summary: str            # 块总结
      t_start: float          # 时间范围
      t_end: float
      key_points: list[str]   # 关键要点（可选）

  这样 Round 1 和 Round 2 都输出这个结构，只是精细度不同。

  Q2：文本消歧——原地修改 vs 保留原文+映射表

  这是个很深的问题。让我从三个角度分析：

  角度 A：LLM 的"逐字遍历"能力

  你的疑虑： LLM 是模糊的，不擅长"逐字遍历，找出具体位置的错字"。

  第一性原理分析：

  LLM 的工作方式是 token 级别的，不是字级别的。当 LLM 看到一段文本时：

  输入文本：
  "张伟说，这个项目很重要。张伟强调..."

  LLM 的 tokenization：
  ["张", "伟", "说", "，", "这", "个", "项", "目", "很", "重", "要", "。", "张",
   "伟", "强", "调", "..."]

  LLM 的处理：
  - 逐 token 生成，同时维护上下文
  - 当看到"张伟"这个 token 序列时，可以识别"这是一个人名"
  - 但 LLM 不会"遍历"，而是"生成"

  关键洞察： LLM 不是"遍历找错"，而是"理解后重新生成"。

  如果你让 LLM 做"原地修改"（即输入原文，输出修正文本），LLM 的行为是：
  1. 理解原文的语义
  2. 根据理解，生成"应该是什么样"的文本
  3. 输出修正后的文本

  这个过程中，LLM 可能会：
  - 修正明显的误识（"瑞斯耐特" → "ResNet"）
  - 但也可能"创意修改"（改变措辞、调整语序）
  - 甚至可能"过度修正"（把正确的也改了）

  所以，原地修改的风险是： 文本不仅被修正，还可能被"改写"。

  角度 B：音轨是根源，文本是衍生物

  你的疑虑： 修正后的文本可能不符合原音轨。

  这是个非常关键的观察。 让我展开：

  现状：
  - 原始 annotations[] 来自 ASR，包含时间戳
  - 每条 annotation 对应音轨的一个时间段

  如果原地修改文本：
  原始：
  annotations[0] = {
    t_start: 0.0,
    t_end: 5.0,
    content: "瑞斯耐特模型很强大"  // ASR 误识
  }

  修正后：
  annotations[0] = {
    t_start: 0.0,
    t_end: 5.0,
    content: "ResNet 模型很强大"  // 修正后
  }

  问题： 时间戳没变，但文本变了。这意味着：
  - 如果后续要生成字幕，字幕会显示"ResNet"
  - 但音轨里实际说的是"瑞斯耐特"
  - 用户看字幕和听音频会不同步（语义上）

  这是个真实的问题。

  角度 C：符号表 + 映射的方案

  更安全的做法是：保留原文，用符号表做映射。

  原始 annotations：
  annotations[0] = {
    t_start: 0.0,
    t_end: 5.0,
    content: "瑞斯耐特模型很强大"
  }

  符号表：
  symbol_table = [
    {
      "raw_form": "瑞斯耐特",
      "corrected_form": "ResNet",
      "category": "term"
    }
  ]

  消歧后的文本（虚拟视图，不修改原文）：
  cleaned_content = "ResNet 模型很强大"

  这样做的好处：
  1. 原文保留：可以追溯 ASR 的原始输出
  2. 映射清晰：符号表明确记录了"什么被改成了什么"
  3. 可逆：如果需要，可以恢复原文
  4. 音轨一致：虽然文本被"虚拟修正"，但时间戳和音轨仍然对应

  但是， 这里有个问题：2b 决策层需要什么输入？

  如果 2b 读的是 cleaned_annotations（虚拟修正后的文本），那么 LLM
  的决策是基于修正后的文本。这是合理的。

  Q3：文本消歧和字典表，是否应该视为两轮中的同一步骤？

  这是最深层的问题。 让我从编译原理的角度重新思考。

  编译原理的类比

  在编译器中：
  - 词法分析：识别 token，构建符号表
  - 语义分析：检查符号表，验证语义正确性

  类比到 2a：
  - Round 1：词法分析 → 识别可能的误识项，构建 symbol_table_candidates
  - Round 2：语义分析 → 确认误识，应用消歧，生成 cleaned_annotations

  从这个角度，它们确实是两个不同的步骤。

  但是，从 LLM 的角度呢？

  关键问题：LLM 能否"分开"做这两件事？

  假设 Round 1 的输出是：
  {
    "symbol_table_candidates": [
      {"raw_form": "瑞斯耐特", "corrected_form": "ResNet"},
      {"raw_form": "爱德姆优化", "corrected_form": "Adam 优化器"}
    ]
  }

  然后 Round 2 的输入是：
  原始文本 + symbol_table_candidates

  请根据候选符号表，确认哪些是真正的误识，并生成消歧后的文本。

  LLM 的行为：
  1. 读原始文本
  2. 查看候选符号表
  3. 对每条候选，判断"是否真的是误识"
  4. 生成消歧后的文本

  这个过程中，LLM 需要：
  - 理解原始文本的语义
  - 理解候选修正的合理性
  - 判断"修正是否会改变语义"

  第一性原理的答案：

  是的，文本消歧和字典表应该视为两个步骤，但它们在 LLM 的处理中是紧密耦合的。

  理由：
  1. 符号表的生成需要"识别"：LLM 需要看原文，识别"哪些词可能是误识"
  2. 消歧的应用需要"确认"：LLM 需要看候选，确认"这个修正是否合理"
  3. 这两个过程不能完全分开：如果分开，LLM 需要重复理解原文

  但是， 从 Prompt 的角度，可以分成两个明确的任务：

  ## Round 1：识别误识候选

  输入：原始文本 + 目标（用户的分析目标）
  任务：识别可能被 ASR 误识的专有名词
  输出：symbol_table_candidates[]

  ## Round 2：确认并应用消歧

  输入：原始文本 + symbol_table_candidates[] + 上一轮的 purpose
  任务：
    1. 确认候选中哪些是真正的误识
    2. 生成消歧后的文本
  输出：symbol_table[] + cleaned_annotations[]

  综合建议

  关于文本消歧的方案

  我的推荐：采用"符号表 + 虚拟消歧"方案，而不是"原地修改"。

  理由：
  1. 保留原文的可追溯性
  2. 避免 LLM 的"创意修改"
  3. 音轨和文本的对应关系不被破坏
  4. 符号表本身就是有价值的产物（可用于后续的字幕生成、术语管理等）

  关于两轮的分工

  Round 1（识别与假设）：
  - 输出：purpose_rough、segments_rough[]、symbol_table_candidates[]

  Round 2（确认与应用）：
  - 输入：Round 1 的输出 + 原始文本
  - 输出：purpose、segments[]、symbol_table[]、cleaned_annotations[]

  关键点： cleaned_annotations[] 不是修改原始
  annotations[]，而是一个新的、基于符号表消歧后的文本视图。

  数据结构建议

  @dataclass
  class Annotation:
      index: int
      t_start: float
      t_end: float
      type: str  # "speech" | "silence"
      content: str  # 原始 ASR 文本
      metadata: dict

  @dataclass
  class CleanedAnnotation:
      index: int
      content: str  # 消歧后的文本
      # 其他字段同 Annotation

  这样 annotations[] 和 cleaned_annotations[] 是平行的，不会相互覆盖。

  关键是在 Prompt 设计中明确分工，让 LLM 知道"先做什么，再做什么"。

  ---
  前向传播的本质

  什么是前向传播？

  在 Transformer 中，前向传播是指：
  输入 tokens → embedding → self-attention → FFN → ... → 输出 logits

  关键特性：
  1. 单向流动：信息从输入流向输出，不能"回头"
  2. Token 序列处理：LLM 逐 token 生成，每个 token 的生成基于之前的所有 tokens
  3. 上下文窗口：LLM 只能看到当前位置之前的所有 tokens（对于 decoder-only 模型）

  但这里有个关键的误解需要澄清：

  ---
  LLM 的"回头看"能力

  你的直觉"我会先读到块边再回去看"，在 LLM 上的对应是什么？

  答案：LLM 可以"回头看"，但方式不同。

  例子：分块和总结

  假设输入文本是：
  [开场介绍项目背景] ... [第一段讲技术方案] ... [第二段讲实验结果] ... [总结]

  当 LLM 生成输出时：

  LLM 的生成过程（逐 token）：

  第 1 个输出 token：生成 "segments"
  第 2 个输出 token：生成 "["
  第 3 个输出 token：生成 "{"
  ...
  第 N 个输出 token：生成 "title": "开场"
  第 N+1 个输出 token：生成 ","
  第 N+2 个输出 token：生成 "summary": "介绍项目背景和目标"
  ...

  在生成"summary"这个 token 时，LLM 的注意力机制做了什么？

  当前位置：正在生成 "summary" 字段的值

  LLM 的 self-attention 会：
  1. 查看已生成的输出（"title": "开场"）
  2. 查看输入文本的所有位置（包括开场部分、中间部分、结尾部分）
  3. 计算"哪些输入位置对生成 summary 最相关"
  4. 基于这些相关位置的信息，生成 summary 的内容

  这就是"回头看"。 LLM 不是"顺序读一遍"，而是在生成每个 token 时，通过 attention
   机制"查看"输入的任何位置。

  ---
  关键洞察：LLM 的处理方式 vs 人类的处理方式

  ┌─────────┬───────────────────────┬───────────────────────────────────────┐
  │  维度   │         人类          │                  LLM                  │
  ├─────────┼───────────────────────┼───────────────────────────────────────┤
  │ 阅读方  │ 顺序读，从头到尾      │ 一次性加载全文，生成时通过 attention  │
  │ 式      │                       │ 查看任何位置                          │
  ├─────────┼───────────────────────┼───────────────────────────────────────┤
  │ 分块识  │ 读到逻辑边界时识别    │ 在生成分块列表时，同时考虑全文的逻辑  │
  │ 别      │                       │ 结构                                  │
  ├─────────┼───────────────────────┼───────────────────────────────────────┤
  │ 块总结  │ 先确定块边，再回头读  │ 在生成块总结时，通过 attention        │
  │         │ 这块                  │ 查看该块的所有内容                    │
  ├─────────┼───────────────────────┼───────────────────────────────────────┤
  │ 效率    │ 需要"回头"，有额外成  │ 一次前向传播完成所有任务，没有"回头"  │
  │         │ 本                    │ 的成本                                │
  └─────────┴───────────────────────┴───────────────────────────────────────┘

  关键区别： LLM 不需要"回头"，因为它在生成时已经可以访问全文的任何位置。

  ---
  这对"分块和总结是否同一轮"的影响

  从 LLM 的角度：

  分块和总结完全可以在同一轮完成，而且效率更高。

  理由：

  1. LLM 一次前向传播就能完成
  输入：原始文本

  LLM 的处理：
  - 加载全文到上下文
  - 生成 segments[] 时，同时生成每个 segment 的 title 和 summary
  - 一次前向传播完成
  2. 不需要"两轮"来做分块和总结
    - Round 1 分块，Round 2 总结 → 需要两次前向传播，浪费计算
    - 一轮同时做 → 一次前向传播，更高效
  3. LLM 的 attention 机制天然支持这种"全局理解"
    - 当生成某个块的 summary 时，LLM 可以同时看到该块的内容和全文的上下文
    - 这比人类的"回头看"更高效

  ---
  那么，为什么还要分两轮？

  如果分块和总结可以一轮完成，为什么我之前建议分两轮？

  答案：不是因为 LLM 的能力限制，而是因为其他原因。

  原因 1：符号表的生成需要两轮

  Round 1 的真正目的是识别误识候选。

  Round 1：
  - 输入：原始文本（含 ASR 误识）
  - 任务：识别可能的误识项
  - 输出：symbol_table_candidates[]

  Round 2：
  - 输入：原始文本 + symbol_table_candidates[]
  - 任务：确认误识，应用消歧，生成 cleaned_annotations[]

  如果一轮完成所有任务：
  一轮：
  - 输入：原始文本（含 ASR 误识）
  - 任务：同时做分块、总结、识别误识、确认误识、生成消歧文本
  - 输出：segments[] + symbol_table[] + cleaned_annotations[]

  这样做的问题：
  - LLM 需要同时做 5 件事，认知负荷很大
  - 如果某个任务出错（比如误识识别不完整），后续的消歧也会出错
  - 难以调试：不知道是识别阶段出错还是确认阶段出错

  原因 2：两轮的"粗→细"策略

  Round 1：快速识别
  - 目标：快速生成初步结果，识别主要问题
  - 输出：粗粒度的 purpose、segments、symbol_table_candidates

  Round 2：精化和确认
  - 目标：基于 Round 1 的结果，精化理解，确认误识
  - 输入：Round 1 的输出作为"已有假设"
  - 输出：精化的 purpose、segments、确认的 symbol_table、cleaned_annotations

  这样做的好处：
  - 如果 Round 1 的结果已经很好，Round 2 可以快速确认
  - 如果 Round 1 的结果有问题，Round 2 可以修正
  - 两轮的 Prompt 可以针对性地优化

  ---
  人类 vs LLM 的处理顺序确实是反的

  人类的自然顺序： 先纠错 → 再理解
  - 因为人类的工作记忆有限，同时处理"错误文本"和"理解语义"很累
  - 先把文本整理通顺，再读一遍理解，认知负荷更低

  LLM 的自然顺序： 先理解 → 再纠错

  原因很简单：纠错本身就依赖理解。

  举个例子：

  ASR 原文："瑞斯耐特模型在图像分类上表现很好"

  LLM 的处理过程：
  1. 看到"瑞斯耐特"，不认识
  2. 看到"模型"、"图像分类"、"表现" → 理解这是在讨论深度学习
  3. 在"深度学习"这个语境下，"瑞斯耐特"的发音对应 → ResNet
  4. 纠正完成

  纠错的前提是知道"应该是什么"，而"应该是什么"来自语境理解。

  你说的"我输入错误的时候，LLM 也能理解"，恰恰证明了这一点：LLM
  先理解了你的意图，然后才能判断哪里是错的。如果 LLM
  只看字面意思，"瑞斯耐特"就是一个合法的中文词组，根本不会被识别为错误。

  ---
  这对 Round 1 / Round 2 的分工意味着什么

  Round 1 应该是：理解 + 识别误识候选

  输入：原始 ASR 文本（含错误）+ 用户目标

  LLM 做的事：
  1. 读全文，理解整体在讲什么（主旨）
  2. 识别叙事结构（分块 + 块总结）
  3. 在理解的过程中，顺便标记"这个词在这个语境下不对"（误识候选）

  输出：purpose + segments[] + symbol_table_candidates[]

  Round 2 应该是：确认误识 + 生成消歧文本

  输入：原始 ASR 文本 + Round 1 的全部输出

  LLM 做的事：
  1. 拿着 symbol_table_candidates，逐条确认"这个修正是否合理"
  2. 生成消歧后的文本视图（cleaned_annotations）

  输出：symbol_table[] + cleaned_annotations[]

  ---
  为什么这个顺序是对的

  编译器类比：

  编译器第一遍（parsing）：
  - 读源代码，构建 AST（理解结构）
  - 同时收集标识符，构建符号表（识别候选）

  编译器第二遍（semantic analysis）：
  - 用符号表做类型检查（确认）
  - 标记错误（纠错）

  对应到 2a：

  Round 1（parsing）：
  - 读 ASR 文本，理解结构（主旨 + 分块）
  - 同时收集可能的误识项（symbol_table_candidates）

  Round 2（semantic analysis）：
  - 用候选符号表做消歧确认
  - 生成修正后的文本视图

  这个顺序之所以对，是因为：

  1. LLM 能从错误文本中提取语义（这是 LLM 的强项）
  2. 但 LLM 不能在没有语义理解的情况下判断"什么是错的"
  3. 所以：理解先行，纠错后行

  2. 易错词候选列表的作用与生命周期

  作用：
  - 作为初步收集的“疑似误识”列表，告诉 Round 2 “这些词可能错了，你帮我确认”
  - 方便 LLM 在确认消歧时，重点关注这些词汇部分，提高纠错精准度

  生命周期：
  - Round 1 结束后只作为输入给 Round 2
  - Round 2 生成确认和过滤后的符号表后，易错词候选列表本身就可丢弃

  ---
  3. 符号表的多重意义

  - 字面映射：
  每条记录提供“原文词形 ↔ 正确词形”的对应关系
  - 语义信息：
  还可以包含词的类别（人名、机构、术语等）
  - 扩展释义（可选）：
  进一步加注释释义信息，如人物身份、观点归属等，有利于：
    - 决策层对话题相关性的理解和选择
    - 审核层理解决策逻辑的合理性（但 MVP 阶段简化不涉及）
    - 后续人工层辅助决策辅导

  ---
  4. 符号表在后续各层的作用

  - 决策层（2b）：
  决策直接用的是消歧过的 cleaned_annotations，符号表间接作用于消歧结果的精准度；
  决策层本身不必直接读符号表，但符号表质量决定消歧文本质量。
  - 审核层（2c）：
  审核层可视为“校验消歧和保留决策效果”，符号表本身并非必需输入，但含释义的符号表
  可以辅助核查关键词覆盖。
  - 人工层（2d）：
  无 LLM 参与，主要依据审核结果和人工反馈。
  但在界面展现时，可以将符号表释义信息用作用户提示，辅助人工理解和反馈。

  ---
  5. 你提到的释义是否必要

  释义不是必须，但较好的设计中建议包含。

  原因：

  - 机器单纯的“词形映射”只能解决纠错，但对“内容理解”“观点归属”无帮助。
  - 例如：“张伟博士”不仅是人名，还有“某观点提出者”的身份信息，辅助 LLM
  做更高级语义推理。
  - 这样的结构化信息更适合在智能层和决策层使用，提升剪辑的精细度和逻辑性。

  已得出的结论

  1. 周目与跨周目的信息流

  结论： 一个周目 = 2a→2b→2c→2d 的完整循环。跨周目时创建新的
  TimelineManifest，将前一周目作为参考上下文。

  理由：
  - 清晰的周期划分便于追踪和调试
  - 新周目的修正基于前一周目的反馈，而非覆盖原数据
  - 符合 Append-only 原则

  ---
  2. 2a 的两轮分工（Round 1 vs Round 2）

  结论：
  - Round 1（理解与候选识别）： 输出 purpose_rough + segments_rough[] +
  symbol_table_candidates[]
  - Round 2（确认与精化）： 输出 purpose + segments[] + symbol_table[] +
  cleaned_annotations[]

  理由：
  - LLM 的处理顺序是"先理解后纠错"，而非人类的"先纠错后理解"
  - Round 1 在理解全文语义的过程中顺便识别误识候选
  - Round 2 基于 Round 1 的理解和候选，做精确的确认和消歧
  - 两轮分工清晰，便于调试和优化

  ---
  3. 分块与块总结的处理

  结论： 分块和块总结应该在同一轮（Round 1）完成，不需要分开。

  理由：
  - LLM 的 attention 机制支持"全局理解"，不需要"回头看"
  - 分块和总结是同一个"理解"过程的两个输出
  - 分开做反而低效（需要两次前向传播）
  - 人类需要"回头看"，但 LLM 不需要

  ---
  4. 文本消歧的方案

  结论： 采用"符号表 + 虚拟消歧"方案，而非"原地修改"。

  理由：
  - 保留原始 ASR 文本的可追溯性
  - 避免 LLM 的"创意修改"（改变措辞、调整语序）
  - 音轨和文本的对应关系不被破坏
  - 符号表本身是有价值的产物

  具体做法：
  - 原始 annotations[] 保持不变
  - 生成新的 cleaned_annotations[]，基于符号表做虚拟消歧
  - 2b 决策层使用 cleaned_annotations[]，而非修改后的 annotations[]

  ---
  5. 人工层（2d）的交互设计

  结论： 采用"4 个文本输入框 + 可视化片段选择（暂时可选）"的设计。

  理由：
  - 引导用户按维度反馈，避免自由文本的模糊
  - 逼用户将模糊需求具体化
  - 代码层可以根据填写情况自动判断"应该修正什么"
  - 符合现有 LLM 能力和用户体验的平衡

  4 个输入框：
  1. 主旨是否有偏差？
  2. 关键词识别错了？
  3. 内容选择意见（多了/少了）
  4. 剪辑时间节点意见（气口、紧凑度等）

  ---
  6. LLM 的"回头看"能力

  结论： LLM 可以通过 attention 机制"回头看"输入的任何位置，不需要"顺序读"。

  理由：
  - Transformer 的 self-attention 允许任意位置之间的交互
  - 生成每个 token 时，LLM 可以查看输入的全部内容
  - 这不是"回头"，而是"全局查看"
  - 因此 LLM 不需要人类那样的"先读到块边再回头看"

  二、理解层（2a）的两轮处理流程

  功能定义：
  2a 分两轮处理，第一轮快速理解和识别，第二轮精化和确认。通过"理解先行、纠错后行
  "的策略，充分利用 LLM 的语义理解能力。

  Round 1 产出物（理解与候选识别）：
  - purpose_rough：粗略的整体视频主旨总结（一句话或简短段落）
  - segments_rough[]：初步的分块结构，每块包含标题、时间区间和简短总结
  -
  symbol_table_candidates[]：可能的易错词/误识词候选列表，含原始词、推测正确词和
   annotation index

  Round 2 产出物（确认与精化）：
  - purpose：精细化、校正好的整体主旨描述
  - segments[]：精细化的分块列表，每块标题和总结更准确、内容更丰富、时间范围确认
  - symbol_table[]：确认且经过筛选的易错词字典，仅保留高置信度条目
  - cleaned_annotations[]：基于确认的字典表对原始 ASR
  文本的虚拟纠正版本，形成人工可读且更准确的文本视图，不修改原始注释

  关键决策点：
  - LLM 的自然处理顺序是"先理解全文语义，再在理解基础上识别和纠错"，不同于人类的
  "先纠错再理解"
  - Round 1 在理解全文逻辑流的过程中顺便识别误识候选，认知负荷单一
  - Round 2 基于 Round 1 的理解和候选，做精确的确认和消歧
  - 易错词候选列表仅为中间产物，Round 2 产出后可丢弃
  - 符号表（字形映射）在纠错完成后价值有限，MVP 不做释义扩展，含释义版本放入
  after-mvp-todo

  三、分块与块总结的同轮处理

  功能定义：
  分块和块总结在 Round 1 同时完成，不需要分开为两个轮次。

  关键决策点：
  - LLM 的 attention
  机制支持"全局理解"，生成块总结时可同时查看该块内容和全文上下文
  - 不需要人类那样的"先读到块边再回头看"的过程
  - 分块和总结是同一个"理解"过程的两个输出，分开做反而低效（需要两次前向传播）
  - 一次前向传播完成所有任务，更高效

  四、文本消歧的虚拟修正方案

  功能定义：
  采用"符号表 + 虚拟消歧"方案，而非"原地修改"原始 ASR 文本。

  实现方式：
  - 原始 annotations[] 保持不变，完整保留 ASR 原始输出
  - 生成新的 cleaned_annotations[]，基于 Round 2 确认的符号表做虚拟消歧
  - 2b 决策层使用 cleaned_annotations[]，而非修改后的 annotations[]

  关键决策点：
  - 保留原始 ASR 文本的可追溯性，便于审计和问题排查
  - 避免 LLM 的"创意修改"（改变措辞、调整语序、过度修正）
  - 音轨和文本的对应关系不被破坏，虽然文本被虚拟修正，但时间戳和音轨仍然对应
  - 符号表本身是有价值的产物，记录了"什么被改成了什么"

  五、决策层（2b）的输入与输出

  功能定义：

  关键决策点：

  十一、LLM 的"回头看"能力与前向传播

  功能定义：
  LLM 可以通过 attention 机制"回头看"输入的任何位置，不需要"顺序读"。

  技术原理：
  - Transformer 的 self-attention 允许任意位置之间的交互
  - 生成每个 token 时，LLM 可以查看输入的全部内容
  - 这不是"回头"，而是"全局查看"

  对设计的影响：
  - LLM 不需要人类那样的"先读到块边再回头看"的过程
  - 分块和总结可以在同一轮完成，无需分开
  - 一次前向传播可以完成多个任务，效率更高

  - speech 条目：keep 必须为 true 或 false

  - 不做局部修正，整体重跑更简单可靠
